## 摘要
无损模型压缩在缓解位精确（bit-exact）大语言模型（LLM）推理服务中的内存和带宽瓶颈方面具有巨大潜力。然而，由于与 GPU 架构存在根本性的设计不匹配，现有方法通常会导致显著的推理减速：在算子（kernel）级别，传统熵编码产生的变长比特流破坏了 SIMT 并行性；在系统级别，解耦的流水线导致了冗余的内存流量。我们提出了 ZipServ，这是一个为高效 LLM 推理协同设计的无损压缩框架。ZipServ 引入了张量核心感知的三重位图编码（TCA-TBE），这是一种新型的定长格式，可实现常数时间内的并行解码，并结合了融合解压-GEMM（ZipGEMM）算子，可将权重实时解压并直接存入 Tensor Core 寄存器。这种“加载压缩数据，计算解压数据”的设计消除了中间缓冲区并最大化了计算强度。实验表明，ZipServ 可将模型大小减少高达 30%，与 NVIDIA 的 cuBLAS 相比实现了高达 2.21 倍的算子级加速，并比 vLLM 的端到端推理平均提速 1.22 倍。ZipServ 是首个在 GPU 上同时提供存储节省和显著推理加速的无损压缩系统。

## Abstract
Lossless model compression holds tremendous promise for alleviating the memory and bandwidth bottlenecks in bit-exact Large Language Model (LLM) serving. However, existing approaches often result in substantial inference slowdowns due to fundamental design mismatches with GPU architectures: at the kernel level, variable-length bitstreams produced by traditional entropy codecs break SIMT parallelism; at the system level, decoupled pipelines lead to redundant memory traffic. We present ZipServ, a lossless compression framework co-designed for efficient LLM inference. ZipServ introduces Tensor-Core-Aware Triple Bitmap Encoding (TCA-TBE), a novel fixed-length format that enables constant-time, parallel decoding, together with a fused decompression-GEMM (ZipGEMM) kernel that decompresses weights on-the-fly directly into Tensor Core registers. This "load-compressed, compute-decompressed" design eliminates intermediate buffers and maximizes compute intensity. Experiments show that ZipServ reduces the model size by up to 30%, achieves up to 2.21x kernel-level speedup over NVIDIA's cuBLAS, and expedites end-to-end inference by an average of 1.22x over vLLM. ZipServ is the first lossless compression system that provides both storage savings and substantial acceleration for LLM inference on GPUs.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：大语言模型（LLM）推理受限于 GPU 显存容量和带宽。虽然量化等“有损压缩”能减小模型，但会损害精度（如长文本推理或安全敏感场景）。“无损压缩”虽能保证位精确（Bit-exact），但现有方案在 GPU 上会导致严重的推理减速。
*   **瓶颈分析**：
    *   **算子级不匹配**：传统熵编码（如 Huffman、ANS）产生变长比特流，导致 GPU 线程间出现严重的控制流分歧（Divergence），破坏了 SIMT 并行性。
    *   **系统级低效**：现有框架通常采用“解耦”流水线，即先在显存中完全解压权重再计算，这增加了冗余的内存读写，抵消了压缩带来的带宽红利。
*   **研究意义**：ZipServ 旨在通过硬件感知的协同设计，使无损压缩不仅能省空间，还能直接加速推理。

### 2. 方法论：核心思想与关键技术
ZipServ 的核心思想是利用 BF16 权重指数位的统计冗余，设计一种固定长度、易于并行解码的格式。
*   **TCA-TBE（张量核心感知三重位图编码）**：
    *   **统计发现**：LLM 权重的指数位分布极度倾斜且具有连续性（Top-7 指数覆盖 >95%）。
    *   **编码设计**：将每个权重映射为 3 位定长码。一个 8×8 的权重块（Tile）被分解为三个 64 位的位图（Bitmaps），分别代表 3 位编码的三个位平面。
    *   **优势**：定长编码允许常数时间内的并行访问，位图结构完美契合 GPU 的位运算指令。
*   **ZipGEMM（融合解压-GEMM 算子）**：
    *   **实时解压**：在执行矩阵乘法时，直接从显存加载压缩权重，在寄存器中通过轻量级位运算（如 `popc` 指令）实时还原 BF16 值。
    *   **寄存器级分发**：解压后的数据直接喂给 Tensor Core 的 `mma.sync` 指令，消除了中间全局内存缓冲区。
*   **阶段感知策略**：
    *   **Decode 阶段**（访存密集型）：使用融合的 ZipGEMM 算子，最大化计算强度。
    *   **Prefill 阶段**（计算密集型）：使用高效的独立解压算子配合标准 cuBLAS，以摊销解压开销。

*   **存储节省**：模型体积平均减少约 30%（BF16 压缩至约 11.3 bit）。
*   **算子加速**：ZipGEMM 在 L40S 上比 cuBLAS 快 1.36×（最高 2.21×），比最快的无损基准 DFloat11 快 5.53×。

### 7. 优点
*   **硬件协同设计**：摒弃了传统的复杂熵编码，改用位图和位运算，极大地提高了 GPU 利用率。
*   **位精确保证**：在不损失任何精度的情况下实现了类似量化的性能收益。
*   **寄存器级融合**：通过“加载压缩-寄存器解压-计算”模式，打破了无损压缩在系统流水线中的性能瓶颈。
*   **前瞻性**：在 Blackwell 架构（RTX 5090）上依然表现优异。

### 8. 不足与局限
*   **小规模算子限制**：在某些形状极小的层（如 8B 模型的 O_proj）上，由于无法充分利用 Split-K 等优化，可能出现轻微减速。

*   **应用范围**：目前主要针对 BF16 格式，虽然理论上可扩展，但对其他格式（如 FP8）的适配尚未在文中深入探讨。
