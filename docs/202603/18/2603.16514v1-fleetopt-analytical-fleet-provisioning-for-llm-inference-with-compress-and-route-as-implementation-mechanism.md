## 摘要
现代大语言模型（LLM）GPU 集群通常针对最坏情况下的上下文长度进行配置，而绝大多数请求从未达到该长度，导致 GPU 容量在空闲的 KV 缓存槽位上被浪费。我们提出了 FleetOpt，这是一个从第一性原理出发的框架：给定工作负载的提示词长度累积分布函数（CDF）和 P99 首字延迟（TTFT）目标，通过分析推导出成本最低的集群配置，并将其应用于实际部署。其分析核心将每个资源池建模为 M/G/c 排队模型，并推导出最低成本集群采用双池架构——即短上下文池和长上下文池——且存在一个最优边界 B*，该边界满足两个资源池之间边际 GPU 成本相等的条件。实现 B* 的根本障碍是“成本悬崖”：在硬性路由步骤中，刚好超过 B* 的请求所消耗的 GPU 容量是刚好低于 B* 请求的 8 到 42 倍（取决于上下文窗口比例），这在结构上抑制了降低边界的动力。“压缩与路由”（C&R）是解决这一障碍的实现机制。网关层的抽取式压缩在推理引擎接收请求之前，将处于边界附近的请求修剪至 B* 以下，从而将硬性的硬件边界转换为从工作负载 CDF 中读取的软件参数。这两个组件统一在 FleetOpt 离线规划器中：给定 CDF 和服务水平目标（SLO），它能在 1 毫秒内返回最优参数 (n_s*, n_l*, B*, gamma*)。在三个生产环境轨迹上，与同构集群相比，该组合框架将 GPU 总成本降低了 6% 至 82%，其中 C&R 根据工作负载类型的不同，在普通资源池路由的基础上额外贡献了 1 到 44 个百分点的降幅。该分析模型通过离散事件模拟器（inference-fleet-sim）进行了验证，在所有资源池和工作负载中，预测的 GPU 利用率误差均小于等于 3%。

## Abstract
Modern LLM GPU fleets are provisioned for worst-case context lengths that the vast majority of requests never approach, wasting GPU capacity on idle KV-cache slots. We present FleetOpt, a framework that starts from first principles: given a workload's prompt-length CDF and a P99 TTFT target, derive the minimum-cost fleet analytically, then deploy it in practice.   The analytical core models each pool as an M/G/c queue and derives that the minimum-cost fleet is a two-pool architecture -- a short-context pool and a long-context pool -- with an optimal boundary B* satisfying an equal marginal GPU cost condition across both pools. The fundamental barrier to achieving B* is the cost cliff: a hard routing step where requests just above B* consume 8x--42x more GPU capacity than requests just below it (depending on the context window ratio), creating a structural disincentive to lower the boundary.   Compress-and-Route (C&R) is the implementation mechanism that resolves this barrier. Gateway-layer extractive compression trims borderline requests below B* before the engine ever sees them, converting the hard hardware boundary into a software parameter read from the workload CDF. The two components are unified in the FleetOpt offline planner: given a CDF and SLO, it returns the optimal (n_s*, n_l*, B*, gamma*) in under 1 ms.   On three production traces, the combined framework reduces total GPU cost by 6--82% versus a homogeneous fleet, with C&R contributing 1--44 percentage points beyond plain pool routing depending on workload archetype. The analytical model is validated against a discrete-event simulator (inference-fleet-sim) with <= 3% error on predicted GPU utilization across all pools and workloads.

*   **研究动机**：现代大语言模型（LLM）支持极长的上下文（如 128K），但生产环境轨迹显示，绝大多数请求（约 80%）长度极短（< 2K）。

*   **成本悬崖（Cost Cliff）**：即使采用长短池分离路由，在边界处仍存在巨大的效率鸿沟。例如，一个请求仅多出一个 token 导致其从短池转入长池，其消耗的 GPU 容量可能瞬间激增 8 到 42 倍。
*   **研究目标**：从第一性原理出发，通过数学建模推导最优集群配置，并引入压缩机制打破硬件边界限制，降低推理成本。

#### 2. 方法论
*   **核心思想**：将集群配置问题转化为一个基于排队论的优化问题，通过“分析建模指导配置，压缩机制辅助实现”的思路，构建双池（短上下文池 $P_s$ 和长上下文池 $P_l$）架构。
*   **关键技术细节**：
    *   **M/G/c 排队模型**：将每个资源池建模为 M/G/c 队列，其中 $c$ 为总 KV 槽位数。利用 Erlang-C 公式和 Kimura 近似法计算 P99 等待时间，确保满足服务等级协议（SLO）。
    *   **最优边界推导**：推导出最优边界 $B^*$ 需满足“等边际 GPU 成本条件”，即向短池增加一个请求的边际成本应等于从长池减少一个请求的边际收益。
    *   **压缩与路由（C&R）**：在网关层引入**抽取式压缩**。针对处于边界附近的“边缘请求”，利用 TextRank、TF-IDF 等算法将其提示词修剪至边界以下，使其能进入高效的短池。
    *   **离线规划器（FleetOpt Planner）**：一个毫秒级的算法，输入工作负载的长度分布（CDF）、到达率和 SLO，输出最优的 GPU 数量、边界值和压缩带宽。

#### 6. 主要结论与发现
*   **显著降本**：相比同构集群，FleetOpt 可降低 **6% 至 82%** 的 GPU 总成本。
*   **压缩的价值**：C&R 机制在普通路由基础上额外贡献了 1.2 到 43.7 个百分点的降幅，尤其在“成本悬崖”剧烈时效果最明显。
*   **边界效应**：通过压缩将硬件的硬边界转化为软件可调参数，使得理论上的最优配置在物理集群中变得可行。
*   **排队论适用性**：在多服务器（Many-server）环境下，集群规模足以忽略排队延迟，配置主要受限于利用率上限。

#### 7. 优点
*   **理论与实践结合**：首次将排队论分析与提示词压缩技术结合，用于 LLM 集群的容量规划。
*   **极高的运行效率**：规划器搜索速度极快，能够应对工作负载分布的动态变化。
*   **硬性显存保证**：通过预设压缩预算，从根本上杜绝了压缩请求导致 KV 缓存溢出（OOM）的风险。
*   **非侵入性**：抽取式压缩在网关层完成，无需修改底层推理引擎。

#### 8. 不足与局限
*   **内容类型限制**：压缩机制仅适用于散文和 RAG 场景，对于代码等结构化内容，由于语义风险被排除在压缩之外。
*   **语义损失风险**：虽然 BERTScore 等指标表现良好，但抽取式压缩本质上是有损的，可能影响极少数复杂任务的精度。
*   **模型依赖性**：分析模型中的硬件常数（如 $W, H$）需针对特定 GPU 型号和模型架构进行校准，通用性受限。
*   **静态分布假设**：规划器依赖于预先获取的 CDF，若工作负载发生剧烈且不可预测的瞬时波动，可能导致配置失效。
