## 摘要
为大语言模型（LLM）推理确定 GPU 集群规模比看起来要困难得多。一些显而易见的问题——例如需要多少个 GPU、选择哪种型号、如何在双池集群中进行划分——都没有闭式解。这些问题取决于完整的 token 长度分布、路由策略以及在重尾工作负载下变得极其复杂的排队动态。现有工具通常针对固定的 GPU 数量优化单个引擎的配置，但没有一个工具能解决上游的规划问题，即应该购买多少个 GPU 以及如何布置它们。inference-fleet-sim 填补了这一空白。它结合了解析式 M/G/c 排队模型与离散事件模拟（DES），旨在寻找能够凭经验满足 P99 TTFT SLO（首字延迟服务水平目标）的最低成本集群配置。该工具包含一个物理感知的 GPU 性能模型，涵盖了 A10G、A100 和 H100，支持单体、双池路由和解耦拓扑，且无需访问真实硬件。我们在源自两个公共工作负载追踪（LMSYS、Azure）和一个合成的智能体密集型追踪的七个集群规划场景中运行了该工具。每个场景都揭示了简单分析会出错的结果——例如正确的划分阈值、最便宜的 GPU 类型、表面空闲的集群是否实际上已失效——并展示了为什么对排队、路由和硬件进行联合模拟是找到这些结果的必要条件。

## Abstract
Sizing a GPU fleet for LLM inference is harder than it looks. The obvious questions -- how many GPUs, which type, where to split a two-pool fleet -- have no closed-form answers. They depend on the full token-length distribution, the routing policy, and queueing dynamics that turn ugly under heavy-tailed workloads. Existing tools optimize per-engine configuration for a fixed GPU count; none of them address the upstream question of how many GPUs to buy and how to arrange them.   inference-fleet-sim fills that gap. It combines analytical M/G/c queueing with discrete-event simulation (DES) to find the minimum-cost fleet configuration that empirically meets a P99 TTFT SLO. It includes a physics-informed GPU performance model covering A10G, A100, and H100 across monolithic, two-pool-routed, and disaggregated topologies, all without requiring access to real hardware. We run the tool on seven fleet-planning scenarios drawn from two public workload traces (LMSYS, Azure) and one synthetic agent-heavy trace. Each one surfaces a result that simple analysis gets wrong -- the right split threshold, the cheapest GPU type, whether an apparently idle fleet is actually broken -- and shows why joint simulation of queueing, routing, and hardware is necessary to find it.

### 2. 方法论
`inference-fleet-sim` 采用了一种**两阶段优化**架构：
*   **第一阶段：解析式快速筛选（Analytical Sweep）**：
    *   利用 **M/G/c 排队模型**和 **Kimura 近似公式**，快速评估成千上万种候选配置（包括不同的划分阈值 $B_{short}$、GPU 数量和型号）。
    *   该阶段在毫秒级内完成，初步筛选出满足利用率上限（如 85%）且理论上符合 SLO 的低成本候选方案。
*   **第二阶段：离散事件仿真验证（DES Verification）**：
    *   对排名靠前的候选方案进行高保真模拟。DES 模拟请求级别的到达、排队和连续批处理（Continuous Batching）动态。
    *   **物理感知 GPU 模型**：通过公式 $t_{iter}(n) = W + H \cdot n$ 建模 GPU 性能，其中 $W$ 是基础计算开销，$H$ 是每并发序列的内存带宽成本。
*   **关键技术细节**：
    *   **可靠性感知**：引入 `node_avail` 参数，根据 GPU 故障率和平均修复时间（MTTR）自动增加冗余节点。
    *   **路由策略**：支持按长度路由、压缩路由、随机路由等多种策略。
    *   **电网灵活性分析**：通过逻辑回归功率模型，评估在电网需求响应（DR）期间降低功耗对 SLO 的影响。

### 6. 主要结论与发现
*   **划分阈值非直观**：最优的短/长请求划分阈值无法直接从 CDF 图表中读出，错误的阈值可能导致成本比不划分更高。
*   **低利用率不代表安全**：在智能体负载下，即使集群利用率仅 30%，也可能因长尾请求导致的“队首阻塞”而违反 SLO。
*   **慢 GPU 可能更省钱**：在特定拓扑下，大量廉价的 A10G 可能比少量昂贵的 H100 更具成本优势。
*   **解耦推理的逆直觉**：在预填充/解码解耦架构中，高性能 GPU（如 H100）应优先用于解码池而非预填充池。
*   **电网响应可行性**：通过降低并发批次大小，集群可以在短时间内削减高达 40% 的功耗而不违反 SLO。

### 7. 优点
*   **填补空白**：首次将排队论与 DES 结合，专门解决 LLM 推理集群的“采购与规模规划”问题。
*   **高效性**：两阶段设计兼顾了搜索速度和验证精度。
*   **实用性强**：考虑了硬件故障、电网功率限制等生产环境中的真实约束。
*   **开源贡献**：作为 vLLM 生态的一部分开源，具有较高的社区价值。

### 8. 不足与局限
*   **泊松流假设**：将泊松流按长度拆分后，子流在严格数学意义上不再是泊松流，虽然 DES 进行了校验，但在极端相关性负载下可能存在偏差。
*   **请求级粒度**：未模拟 token 级的抢占、调度重排或投机采样等微观行为，精度略低于 Vidur 等工具。
*   **线性模型简化**：GPU 性能采用线性 Roofline 模型，未显式建模 FlashAttention 优化、量化加速或多节点通信开销（需手动调整参数）。
*   **功率模型限制**：功率预测基于 H100 的拟合曲线，对于其他型号或极低批次情况下的准确性可能受限。
