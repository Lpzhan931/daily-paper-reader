## 摘要
智能体工作流由一系列相互依赖的大语言模型（LLM）调用组成，已成为现代 AI 系统中的主流工作负载。由于推测性执行和并行探索，这些工作流在重叠的提示词和中间结果中表现出大量的冗余。现有的 LLM 推理服务系统（如 vLLM）专注于优化单个推理调用，而忽略了跨调用的依赖关系，导致了显著的效率低下。本文从数据系统的视角重新思考了 LLM 和智能体推理服务，并引入了 Helium。Helium 是一个感知工作流的推理服务框架，它将智能体工作负载建模为查询计划，并将 LLM 调用视为一等算子。Helium 集成了主动缓存和感知缓存的调度，以最大限度地提高提示词、KV 状态和工作流之间的复用。通过这些技术，Helium 将经典的查询优化原则与 LLM 推理服务相结合，在各种工作负载上相比最先进的智能体推理服务系统实现了高达 1.56 倍的加速。我们的结果表明，跨工作流的端到端优化对于构建可扩展且高效的基于 LLM 的智能体至关重要。

## Abstract
Agentic workflows are composed of sequences of interdependent Large Language Model (LLM) calls, and they have become a dominant workload in modern AI systems. These workflows exhibit extensive redundancy from overlapping prompts and intermediate results due to speculative and parallel exploration. Existing LLM serving systems, such as vLLM, focus on optimizing individual inference calls and overlook cross-call dependencies, leading to significant inefficiencies. This paper rethinks LLM and agent serving from a data systems perspective and introduces Helium, a workflow-aware serving framework that models agentic workloads as query plans and treats LLM invocations as first-class operators. Helium integrates proactive caching and cache-aware scheduling to maximize reuse across prompts, KV states, and workflows. Through these techniques, Helium bridges classic query optimization principles with LLM serving, achieving up to 1.56x speedup over state-of-the-art agent serving systems on various workloads. Our results demonstrate that end-to-end optimization across workflows is essential for scalable and efficient LLM-based agents.

以下是对该论文的结构化总结：

### 1. 核心问题与研究动机
*   **核心问题**：现代 AI 智能体工作流由一系列相互依赖的 LLM 调用组成，具有高度的**计算冗余**（如重叠的提示词前缀）和**复杂的依赖关系**。
*   **研究背景**：现有的 LLM 推理引擎（如 vLLM）主要针对单次推理调用进行优化（如连续批处理、PagedAttention），但对跨调用的工作流结构缺乏感知，导致在处理复杂的智能体任务时，无法有效复用 KV 缓存和中间结果，造成资源浪费和高延迟。
*   **研究动机**：借鉴数据库查询优化的原则，将智能体工作流视为“查询计划”，将 LLM 调用视为“算子”，通过全局优化来消除冗余。

### 2. 方法论：Helium 框架
Helium 引入了工作流感知的推理服务层，核心技术包括：
*   **工作流建模（DSL & DAG）**：使用 Python DSL 将工作流定义为有向无环图（DAG），将 LLM 调用建模为一等算子（First-class Operators）。
*   **逻辑计划优化**：
    *   **算子剪枝与公共子图消除（CSE）**：移除不必要的计算分支，合并相同的子图。
    *   **提示词缓存替换**：对于确定性算子，若输入相同则直接从全局缓存中获取结果（CacheFetch），绕过昂贵的 LLM 推理。
*   **模板化基数树（Templated Radix Tree, TRT）**：一种创新的数据结构，用于捕获提示词的静态前缀结构、动态占位符及其依赖关系，为调度提供全局视图。
*   **主动缓存管理（Proactive Caching）**：不同于现有的被动缓存，Helium 在执行前识别静态前缀并预热 KV 缓存，减少 Prefill 阶段的重复计算。
*   **感知缓存的成本调度算法**：基于 TRT 和成本模型，通过贪心算法优化算子在多 GPU 上的分配和执行顺序，最大化 KV 缓存的复用率并平衡负载。

### 3. 实验设计
*   **数据集/场景**：
    *   **微基准测试**：涵盖 Map-Reduce、多智能体辩论（Debate）、自我反思（Reflection）、迭代改进（Iterative）、并行链（Parallel）等 5 种原始模式。
    *   **端到端场景**：构建了一个复杂的“金融交易（Trading）”工作流，包含 19 个智能体和 88 个 LLM 算子。
    *   **数据源**：MMLU、TAT-QA、Amazon Reviews 以及自定义金融数据集。
*   **对比方法（Baselines）**：
    *   **推理引擎**：原生 vLLM。
    *   **编排框架**：LangGraph、AgentScope。
    *   **学术前沿系统**：Parrot（感知语义变量）、KVFlow（感知工作流的 KV 缓存管理）、OpWise（算子级批处理）。

### 4. 资源与算力
*   **硬件环境**：一台配备 AMD EPYC 9554 64 核处理器和 **2 块 94GB NVIDIA H100 NVL GPU** 的服务器。
*   **模型**：主要使用 **Qwen3-8B/14B/32B** 和 **Llama-3.1-8B**。
*   **软件基础**：基于 vLLM v0.16.0 构建。

### 5. 实验数量与充分性
*   **实验规模**：进行了多维度的评估，包括 5 种原始工作流的性能对比、1 个复杂端到端场景的深度分析、消融实验（验证剪枝、缓存、调度各自的贡献）、以及针对批大小、分支数、模型大小、内存限制的敏感性分析。
*   **充分性评价**：实验设计非常全面且客观。不仅对比了端到端延迟，还分析了 KV 缓存命中率、调度算法的近优性（与 MILP 最优解对比）以及系统开销（CPU/内存占用），证明了其在不同负载下的鲁棒性。

### 6. 主要结论与发现
*   **性能提升**：Helium 在原始工作流上比最先进系统快 **1.56 倍**，在复杂金融工作流中实现 **1.34 倍** 加速，相比原生 vLLM 提升可达 **39.5 倍**。
*   **缓存效率**：通过主动调度，KV 缓存命中率显著提升（例如在 Qwen3-8B 上比其他系统高出约 30%）。
*   **调度优势**：Helium 的全局调度算法能有效缓解尾部延迟（P95 延迟显著降低），且调度开销极低（毫秒级）。

### 7. 优点
*   **视角独特**：成功将成熟的数据库查询优化理论（如算子化、CSE、成本模型）引入 LLM 推理领域。
*   **主动性**：从“被动等待请求”转变为“主动分析工作流结构”，通过预热和全局调度解决了智能体工作流的结构性冗余。
*   **系统性强**：不仅有理论模型，还提供了完整的 DSL 实现和针对 vLLM 的底层优化。

### 8. 不足与局限
*   **动态控制流限制**：目前的 DAG 模型对高度动态的控制流（如基于运行时输出的无限循环或动态分支）支持有限。
*   **外部 API 依赖**：对于包含大量外部工具调用（如搜索、数据库查询）的智能体，外部延迟的不可预测性可能会干扰其成本模型的准确性。
*   **同质模型假设**：实验主要假设工作流使用相同的基座模型，对于混合使用多种不同尺寸模型的异构场景讨论较少。
