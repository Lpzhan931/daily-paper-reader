## 摘要
大语言模型的上下文窗口并非内存，而是 L1 缓存：一种小容量、高速且昂贵的资源，而目前该领域却将其视为整个内存系统。这里没有 L2 缓存，没有虚拟内存，也没有分页机制。每一个工具定义、每一条系统提示词以及每一个过时的工具结果，在整个会话生命周期内都会占用上下文。其结果是可衡量的：在 857 个生产会话和 445 万个有效输入 token 中，有 21.8% 属于结构性浪费。我们提出了 Pichay，一个针对 LLM 上下文窗口的请求分页系统。Pichay 作为客户端与推理 API 之间的透明代理实现，它介入消息流以驱逐陈旧内容，在模型重新请求已驱逐材料时检测缺页异常，并根据缺页历史记录固定工作集页面。在 140 万次模拟驱逐的离线回放中，缺页率为 0.0254%。在超过 681 轮的生产环境部署中，该系统将上下文消耗降低了高达 93%（从 5,038KB 降至 339KB）；在极端持续压力下，系统仍能运行，但会出现预期的抖动病态现象，即已驱逐内容被反复调入。核心观察是，该领域面临的问题（如上下文限制、注意力退化、成本扩展、跨会话状态丢失）本质上都是披着不同外衣的虚拟内存问题。解决方案已经存在：工作集理论（Denning, 1968）、请求分页、缺页驱动的替换策略，以及具有多个驱逐管理层级的内存层级结构。我们描述了 LLM 系统完整内存层级（从 L1 到持久化存储）的架构，报告了在生产中部署的前三个层级（L1 驱逐、L2 缺页驱动固定、L3 模型发起的对话压缩），并将跨会话内存确定为最后的待开发领域。

## Abstract
The context window of a large language model is not memory. It is L1 cache: a small, fast, expensive resource that the field treats as the entire memory system. There is no L2, no virtual memory, no paging. Every tool definition, every system prompt, and every stale tool result occupies context for the lifetime of the session. The result is measurable: across 857 production sessions and 4.45 million effective input tokens, 21.8% is structural waste.   We present Pichay, a demand paging system for LLM context windows. Implemented as a transparent proxy between client and inference API, Pichay interposes on the message stream to evict stale content, detect page faults when the model re-requests evicted material, and pin working-set pages identified by fault history. In offline replay across 1.4 million simulated evictions, the fault rate is 0.0254%. In live production deployment over 681turns, the system reduces context consumption by up to 93% (5,038KB to 339KB); under extreme sustained pressure, the system remains operational but exhibits the expected thrashing pathology, with repeated fault-in of evicted content.   The key observation is that the problems the field faces, such as context limits, attention degradation, cost scaling, lost state across sessions, are virtual memory problems wearing different clothes. The solutions exist: working set theory (Denning, 1968), demand paging, fault-driven replacement policies, and memory hierarchies with multiple eviction-managed levels. We describe the architecture of a full memory hierarchy for LLM systems (L1 through persistent storage), report on the first three levels deployed in production use (L1 eviction, L2 fault-driven pinning, L3 model-initiated conversation compaction), and identify cross-session memory as the remaining frontier.

*   **核心矛盾**：论文指出，当前的 LLM 领域将“上下文窗口”误视为整个内存系统，而实际上它更接近计算机体系结构中的 **L1 缓存**（小容量、高速、昂贵）。

*   **研究动机**：作者认为，解决上下文限制、成本飙升和注意力退化问题的关键不在于无限扩大 L1 窗口（如 1M 或 10M tokens），而在于引入成熟的**虚拟内存（Virtual Memory）**和**内存层级（Memory Hierarchy）**管理机制。

### 2. 核心方法论
作者开发了名为 **Pichay** 的系统，其核心思想是将操作系统中的“请求分页”概念引入 LLM 上下文管理：
*   **透明代理架构**：Pichay 作为一个 HTTP 代理运行在客户端和推理 API 之间，无需修改模型或客户端代码即可拦截并修改消息流。
*   **垃圾回收 (GC) 与 分页 (Paging)**：
    *   **GC**：针对不可重用的临时输出（如 Bash 命令结果）。
    *   **Paging**：针对可寻址内容（如文件读取结果）。被驱逐的内容会被替换为一个微小的“检索句柄”（Retrieval Handle），如 `[Paged out: Read file.py...]`。
*   **缺页驱动固定 (Fault-Driven Pinning)**：当模型通过句柄重新请求已驱逐的内容时，系统记录一次“缺页异常”（Page Fault），并将该内容标记为“核心工作集”进行固定，防止再次驱逐。
*   **协作式内存管理**：
    *   **幻影工具 (Phantom Tools)**：注入模型可见但框架不可见的工具，允许模型主动释放内存或请求调入缓存。
    *   **清理标签 (Cleanup Tags)**：模型可在输出中嵌入指令，自主决定压缩对话历史（L3 层级管理）。
*   **反转成本模型 (Inverted Cost Model)**：不同于传统 OS（保留内存免费，缺页昂贵），LLM 环境下保留 token 极贵，而重新读取（缺页）相对便宜。因此，Pichay 采取了极其激进的驱逐策略。

*   **数据集/场景**：
    *   **生产语料库**：收集了 857 个真实的生产会话（来自 Claude Code），包含 54,170 次 API 调用，总计 44.5 亿有效输入 token。

*   **Benchmark 与对比方法**：

### 4. 资源与算力
*   **模型使用**：实验主要基于 Anthropic 的 **Claude 3 Opus**、**Sonnet 3.5/4** 和 **Haiku 4.5**。

### 5. 实验数量与充分性
*   **实验规模**：

### 6. 主要结论与发现
*   **浪费严重**：生产环境中 21.8% 的 token 属于结构性浪费（未使用工具定义占 11%，重复内容 2.2%，陈旧工具结果 8.7%）。
*   **放大效应**：工具结果在上下文中的中位放大因子为 84.4x，意味着每 1KB 的输出会被重复处理 84 次以上。
*   **性能提升**：Pichay 在生产中将上下文占用降低了最高 93%，且在 LLM 评审中，由于减少了无关信息的干扰，模型在某些情况下的表现甚至优于全量上下文（注意力更集中）。
*   **病态抖动**：当工作集超过物理限制时，系统会出现类似 OS 的“抖动”（Thrashing），即频繁地驱逐和重新调入内容，导致成本激增。

### 7. 优点
*   **架构洞察力**：首次将 LLM 上下文管理严谨地映射到操作系统虚拟内存理论，提供了清晰的层级模型（L1-L4）。
*   **实用性极强**：作为透明代理实现，无需模型微调或 API 变更，具有即插即用的特性。
*   **协作创新**：提出的“协作式内存管理”利用了 LLM 的智能，让模型参与自身的缓存管理，这是传统硬件内存管理无法实现的。

### 8. 不足与局限
*   **样本偏差**：语料库主要基于单一用户和单一工具（Claude Code），在其他类型的 Agent 或通用对话中的表现尚待验证。
*   **前瞻性依赖**：目前的驱逐策略（FIFO）虽然简单有效，但在处理具有长程依赖的任务时，可能会导致较高的缺页成本。
*   **L3/L4 验证不足**：论文重点评估了 L1 和 L2 层级，对于 L3（对话压缩）和 L4（跨会话持久化）的评估尚处于初步阶段。
*   **成本风险**：在极端情况下，频繁的缺页会导致额外的 API 调用轮次，从而产生意料之外的金钱成本。
