## 摘要
大规模机器学习工作负载日益依赖多 GPU 系统，然而其性能往往受限于一个被忽视的组件：CPU。通过对现代大语言模型（LLM）推理和服务工作负载的详细研究，我们发现多 GPU 性能下降通常并非因为 GPU 达到饱和，而是因为 CPU 无法使 GPU 保持繁忙。在 CPU 分配受限的情况下，系统表现出内核启动延迟、通信停滞以及分词延迟增加等症状，导致即使在 GPU 资源充足时也会出现严重的 GPU 利用率不足。本研究对多 GPU LLM 推理中 CPU 引起的性能下降进行了系统分析。我们证明，即使在采用进程级隔离和 CUDA Graphs 等现代 GPU 端优化的服务栈中，这些瓶颈依然存在。由于额外 CPU 核心的边际成本相对于 GPU 实例定价较低，我们的评估表明，增加 CPU 核心数量能以极小的额外成本显著提升性能和稳定性。在中等服务负载下，我们观察到 CPU 资源匮乏的配置经常发生超时，而提供充足的 CPU 资源可恢复响应速度，并在不同配置下将首字延迟（TTFT）降低 1.36 至 5.40 倍，且无需增加 GPU。本研究表明，CPU 资源配置是多 GPU LLM 推理配置中的关键因素，有助于防止控制侧瓶颈。

## Abstract
Large-scale machine learning workloads increasingly rely on multi-GPU systems, yet their performance is often limited by an overlooked component: the CPU. Through a detailed study of modern large language model (LLM) inference and serving workloads, we find that multi-GPU performance frequently degrades not because GPUs are saturated, but because CPUs fail to keep the GPUs busy. Under limited CPU allocations, systems exhibit symptoms such as delayed kernel launch, stalled communication, and increased tokenization latency, leading to severe GPU underutilization even when ample GPU resources are available. This work presents a systematic analysis of CPU-induced slowdowns in multi-GPU LLM inference. We show that these bottlenecks persist even in serving stacks that employ process-level separation and modern GPU-side optimizations such as CUDA Graphs. Since the marginal cost of additional CPU cores is small relative to GPU instance pricing, our evaluation indicates that increasing the number of CPU cores can substantially improve performance and stability at minimal additional cost. Under moderate serving load, we observe that CPU-starved configurations frequently time out, while providing adequate CPU resources restores responsiveness and reduces time-to-first-token (TTFT) latency by 1.36-5.40x across configurations, all without requiring additional GPUs. This work shows that CPU provisioning is a crucial factor in multi-GPU LLM inference configuration, helping prevent control-side bottlenecks.

#### 1. 核心问题与研究动机
*   **核心问题**：在多 GPU 大语言模型（LLM）推理场景中，业界普遍认为 GPU 是唯一的性能瓶颈。然而，本文指出 **CPU 往往是隐形的瓶颈**。当 CPU 资源分配不足时，会导致内核启动延迟、通信停滞和分词（Tokenization）变慢，从而使昂贵的 GPU 资源处于闲置状态。
*   **研究动机**：在云端或共享集群中，用户通常手动指定 CPU 和 GPU 的比例。由于缺乏指导，CPU 往往被低估和欠分配。作者旨在系统性地量化 CPU 对多 GPU 推理性能的影响，并揭示其背后的物理机制。

#### 2. 核心方法论
论文通过对现代 LLM 推理栈（以 vLLM 为例）的深度剖析，识别并分析了三个关键的 CPU 密集型环节：
*   **分词与预处理**：分词操作完全在 CPU 上运行，且随着序列长度和并发请求增加，其计算开销呈线性增长，直接占据了首字延迟（TTFT）的很大比例。
*   **内核启动与同步控制**：CPU 负责向 GPU 发送指令。在多 GPU 环境下，NCCL 等通信库依赖屏障同步（Barrier Synchronization）。如果 CPU 因资源竞争延迟了某一个 GPU 的内核启动，会导致整个 GPU 组陷入忙等（Busy-wait）。
*   **共享内存广播竞争**：分析了 vLLM V1 架构中的“1 写 N 读”共享内存模式。即使采用无锁设计，CPU 在轮询（Polling）标志位时产生的竞争也会随张量并行（TP）度的增加而加剧。

#### 3. 实验设计
*   **模型与负载**：使用了 Llama 3.1 8B 和 Qwen 2.5 14B 模型。
*   **实验场景**：
    *   **攻击者-受害者模型**：通过发送高频率的背景请求（Attacker）产生 CPU 负载，测量目标请求（Victim）的延迟。
    *   **真实集群日志分析**：分析了某大学研究与教学集群中 2024 年全年的 465 万条作业记录，统计 CPU 与 GPU 的实际分配比例。
*   **Benchmark 与对比**：
    *   对比了不同 CPU 核心分配（从最小的 `#GPUs + 1` 到充足的 `8 × #GPUs`）。
    *   测量指标包括 TTFT、每秒请求数（RPS）、CPU/GPU 利用率轨迹。

#### 4. 资源与算力
实验涵盖了多种高性能硬件配置：
*   **GPU 型号**：NVIDIA H100 (Hopper)、H200 (Hopper) 以及 RTX Pro 6000 (Blackwell 架构)。
*   **节点配置**：每节点 8 个 GPU，通过 NVLink 4.0 或 PCIe 5.0 连接。
*   **CPU**：Intel Xeon Platinum 8480CL (64 核) 或 Intel Xeon 6737P。
*   **软件环境**：vLLM v0.11.1，启用了 CUDA Graphs、Chunked Prefill 等现代优化技术。

#### 5. 实验数量与充分性
*   **实验规模**：论文在三种不同的 GPU 架构上进行了测试，涵盖了 4 GPU 和 8 GPU 的配置，并测试了从 1.8k 到 114k 不等的序列长度。
*   **充分性**：实验设计较为全面，不仅有合成负载下的性能测试，还结合了真实世界的集群分配数据，证明了研究问题的普遍性。通过消融实验展示了 CPU 核心数对延迟的直接影响，逻辑闭环较强。

#### 6 主要结论与发现
*   **显著提速**：在不增加 GPU 的情况下，仅通过增加 CPU 核心分配，可将 TTFT 延迟降低 **1.36 至 5.40 倍**。
*   **超时风险**：在 CPU 资源匮乏（如 4 GPU 仅配 5 核）时，系统在高负载下会频繁发生超时（Timeout），而增加 CPU 资源可恢复系统稳定性。
*   **GPU 闲置**：CPU 竞争会导致 GPU 利用率大幅波动。即使 GPU 算力充足，若 CPU 无法及时调度内核，GPU 也会因等待同步信号而浪费电力。
*   **普遍欠分配**：日志分析显示，大量用户为 8 个 H100 仅申请极少数 CPU 核心，中位数 CPU/GPU 比例仅为 1-2，远低于维持高性能所需的水平。

#### 7. 优点
*   **视角独特**：挑战了“LLM 推理只看 GPU”的固有认知，量化了控制侧瓶颈。
*   **实战指导性强**：为云服务用户提供了明确的配置建议，指出增加 CPU 核心是极具性价比（成本仅增加约 1.5%）的优化手段。
*   **根因分析深入**：不仅展示了现象，还深入到 Linux 共享内存机制和 NCCL 同步机制层面解释了瓶颈来源。

#### 8. 不足与局限
*   **硬件覆盖**：主要集中在 NVIDIA GPU 和 Intel CPU，未测试 AMD (ROCm) 或 ARM 架构（如 Grace Hopper）下的表现，后者可能具有不同的内核启动特性。
*   **软件依赖**：实验高度依赖 vLLM 的多进程架构，虽然具有代表性，但对于 TensorRT-LLM 等其他框架的泛化性有待进一步验证。
*   **训练负载缺失**：研究重点在于推理，虽然提及了训练负载，但未进行深入的实验表征。
*   **超线程影响**：实验中关闭了 SMT（超线程），而云端实例通常提供的是 vCPU，实际环境中的干扰可能比实验测得的更复杂。
