## 摘要
大语言模型（LLM）生成仍然昂贵，因为自回归解码每生成一个新词元就调用一次模型。推测解码通过一次草拟多个词元并用目标模型一步验证来降低成本，但其加速效果取决于被接受的草拟词元数量。无参数草拟源可以在结构和智能体工作负载中以低成本提出长续写，但在一个生成步骤中看起来有希望的缓存匹配可能在下一步中收益很低。我们提出了混合验证解码，它在验证前预测缓存草稿的接受长度，并使用这一收益估计在缓存验证和基于模型的草拟器之间进行选择。在三个LLM和十六个数据集上，混合验证解码在智能体工作流中尤其有效，在每个设置中均优于EAGLE3，平均加速2.73倍。我们的分析展示了提示结构如何创造缓存机会，高收益缓存草稿如何集中在草稿空间的一小部分，以及收益引导的选择如何减少顺序解码工作，这表明运行时草稿选择是推测解码一个有前景的方向。

## Abstract
Large Language Model (LLM) generation remains expensive because autoregressive decoding calls the model once for each new token. Speculative decoding reduces this cost by drafting multiple tokens and verifying them with the target model in one step, but its speedup depends on how many drafted tokens are accepted. Parameter-free draft sources can propose long continuations at low cost in structured and agentic workloads, yet a cache match that looks promising at one generation step may have low payoff at the next. We propose Hybrid Verified Decoding, which predicts the accepted length of a cache draft before verification and uses this payoff estimate to choose between cache verification and a model-based drafter. Across three LLMs and sixteen datasets, Hybrid Verified Decoding is especially effective on agentic workflows, where it outperforms EAGLE3 in every setting with a 2.73x average speedup. Our analysis shows how prompt structure creates cache opportunities, how high-payoff cache drafts concentrate in a small part of the draft space, and how payoff-guided selection reduces sequential decoding work, pointing to runtime draft selection as a promising direction for speculative decoding.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：大型语言模型（LLM）的自回归生成成本高昂，每个新token都需要一次前向传播。推测解码通过先由轻量级源草拟多个token，再由目标模型一次性验证来加速，但其加速效果高度依赖于每步草拟token中被接受的数量。
- **关键挑战**：无参数草稿源（如缓存匹配）能以低成本提出长续写，但缓存草稿的“回报”（即被接受的长度）在不同解码状态下极不稳定。一个步骤中看似有希望的缓存匹配，在下一步可能只被接受很短的前缀，导致验证开销浪费。现有方法（如EAGLE3）虽然提高了草稿可靠性，但未能灵活处理缓存草稿的动态回报问题。
- **解决思路**：论文提出**混合验证解码（Hybrid Verified Decoding）**，在**验证之前**预测当前缓存草稿的接受长度（即回报估计），并根据此估计动态决定是验证缓存草稿还是回退到基于模型的草稿器（如EAGLE3），从而在运行时分配验证资源，最大化加速。

## 2. 方法论：核心思想、关键技术细节与算法流程
- **核心思想**：将草稿选择建模为**状态依赖的回报预测问题**。在每个解码步骤，先获取缓存草稿，然后使用一个轻量级预测器估算其被目标模型接受的长度。如果预测值足够高（超过阈值 τ），则提交缓存草稿进行验证；否则，使用更可靠的基于模型的草稿器（如EAGLE3）。
- **关键技术细节**：
    1.  **草稿源**：实例化无参数草稿源为**后缀缓存（Suffix Cache）**，基于当前前缀匹配历史上下文来生成候选续写。
    2.  **回报预测器（Payoff Predictor）**：
        - **特征（ϕ_t）**：仅使用验证前可获取的信号，包括：缓存匹配特征（匹配长度、分数等）、解码状态（提示长度、已生成长度等）、近期历史（最近缓存草稿的选择率和接受长度）、草稿结构（空白、标点等词元指示符）。
        - **模型**：一个轻量级多层感知机（MLP），2隐藏层，每层256个神经元，ReLU激活。
        - **训练**：通过回放（Replay）目标模型生成轨迹来获得标签。对于每个解码位置，计算缓存草稿与实际继续生成的token匹配的最大前缀长度作为标签 y_t。训练目标是平方误差回归。
    3.  **决策阈值**：阈值 τ 设置为 6（即预测接受长度 ≥ 6 时才验证缓存草稿）。该阈值通过分析确定。
- **算法流程**：
    1.  初始化前缀 x，后缀缓存 C，历史摘要 h。
    2.  在每一步解码：
        a.  获取缓存草稿 d_c = D_C(x; C)。
        b.  计算特征 ϕ，预测回报 ŷ = g_θ(ϕ)。
        c.  如果 ŷ ≥ τ：
            -   选择缓存草稿进行验证。
        d.  否则：
            -   使用模型草稿器 D_m(x) 生成草稿。
        e.  对选择的草稿执行标准推测验证： (a, z) = Verify(x, d; M)。
        f.  将接受的块 z 追加到输出，更新缓存和历史。
