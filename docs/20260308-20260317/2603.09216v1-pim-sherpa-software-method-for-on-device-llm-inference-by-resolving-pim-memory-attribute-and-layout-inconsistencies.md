## 摘要
大语言模型（LLM）的端侧部署正在移动和边缘平台迅速普及。LLM 推理包含计算密集型的预填充（prefill）阶段和内存带宽密集型的解码（decode）阶段，学术界和工业界普遍认为解码阶段非常适合存内计算（PIM）。然而，实际支持 PIM 的系统在这两个阶段之间面临两个障碍：一是内存属性不一致，即预填充阶段倾向于将权重放置在可缓存区域以供复用，而解码阶段则需要权重位于不可缓存区域以可靠地触发 PIM；二是主机友好型布局与 PIM 感知型布局之间的权重布局不一致。为了解决这些问题，我们引入了 PIM-SHERPA，这是一种纯软件方法，通过解决 PIM 内存属性和布局不一致性来实现高效的端侧 LLM 推理。PIM-SHERPA 提供了两种方法：DRAM 双缓冲（DDB），它将单一的 PIM 感知权重保留在不可缓存区域，同时将下一层的交织（swizzled）权重预取到小型可缓存缓冲区中；以及带有交织内存复制的在线权重重排（OWR），它在执行 GEMM 之前立即进行按需的交织内存复制。与基准 PIM 仿真系统相比，PIM-SHERPA 在 Llama 3.2 模型上实现了约 47.8 - 49.7% 的内存容量节省，同时保持了与理论最大值相当的性能。据我们所知，这是首项识别出内存属性不一致性并在产品级支持 PIM 的系统上提出有效解决方案的工作。

## Abstract
On-device deployments of large language models (LLMs) are rapidly proliferating across mobile and edge platforms. LLM inference comprises a compute-intensive prefill phase and a memory bandwidth-intensive decode phase, and the decode phase has been widely recognized as well-suited to processing-in-memory (PIM) in both academia and industry. However, practical PIM-enabled systems face two obstacles between these phases, a memory attribute inconsistency in which prefill favors placing weights in a cacheable region for reuse whereas decode requires weights in a non-cacheable region to reliably trigger PIM, and a weight layout inconsistency between host-friendly and PIM-aware layouts. To address these problems, we introduce \textit{PIM-SHERPA}, a software-only method for efficient on-device LLM inference by resolving PIM memory attribute and layout inconsistencies. PIM-SHERPA provides two approaches, DRAM double buffering (DDB), which keeps a single PIM-aware weights in the non-cacheable region while prefetching the swizzled weights of the next layer into small cacheable buffers, and online weight rearrangement with swizzled memory copy (OWR), which performs the on-demand swizzled memory copy immediately before GEMM. Compared to a baseline PIM emulation system, PIM-SHERPA achieves approximately 47.8 - 49.7\% memory capacity savings while maintaining comparable performance to the theoretical maximum on the Llama 3.2 model. To the best of our knowledge, this is the first work to identify the memory attribute inconsistency and propose effective solutions on product-level PIM-enabled systems.

### 1. 核心问题与整体含义（研究动机和背景）
论文聚焦于**端侧（On-device）大语言模型（LLM）推理**在结合**存内计算（PIM）**技术时面临的系统级挑战。
*   **研究动机**：LLM 推理分为预填充（Prefill）和解码（Decode）两个阶段。Prefill 是计算密集型的（GEMM），需要权重驻留在**可缓存（Cacheable）**区域以实现数据复用；而 Decode 是带宽密集型的（GEMV），适合 PIM 加速，但 PIM 触发要求权重必须在**不可缓存（Non-cacheable）**区域，否则请求会被 CPU 缓存拦截。
*   **核心冲突**：
    1.  **内存属性不一致**：Prefill 要缓存，Decode 禁缓存。
    2.  **权重布局不一致**：主机 CPU 偏好交织布局（Host-friendly），而 PIM 偏好按 Bank 连续存储的布局（PIM-aware）。
*   **现有方案局限**：目前的做法通常是“权重复制”（内存占用翻倍）或“修改硬件控制器”，这在内存受限的移动设备上难以实施。

### 2. 方法论：PIM-SHERPA 核心思想与技术细节
PIM-SHERPA 是一种**纯软件**方法，旨在不修改硬件的前提下，通过动态管理内存来消除上述不一致性。
*   **核心思想**：仅在内存中保留一份 PIM-aware 布局的权重（存放在不可缓存区），在 Prefill 阶段通过高效的软件手段将其动态转换并搬运至小型可缓存缓冲区。
*   **关键技术**：
    *   **交织内存拷贝（Swizzled Memory Copy, SMC）**：在将权重从不可缓存区拷贝到可缓存区时，同步完成从 PIM-aware 到 Host-friendly 的布局转换。
    *   **DRAM 双缓冲（DDB）**：利用 LLM 逐层计算的特性，在计算当前层 GEMM 的同时，开启后台线程将下一层的权重通过 SMC 预取到另一个小型可缓存缓冲区中，从而掩盖重排延迟。
    *   **在线权重重排（OWR）**：针对长序列场景，在每层 GEMM 执行前即时进行串行 SMC。虽然有额外开销，但在计算占主导的长序列中，该开销占比极小。
*   **算法流程**：系统初始化 PIM 布局权重 -> Prefill 开始 -> 启动 DDB/OWR 线程 -> SMC 转换布局并存入 Cacheable Buffer -> 主机执行 GEMM -> 进入 Decode 阶段 -> 直接在 Non-cacheable 区触发 PIM 执行 GEMV。

### 3. 实验设计
*   **模型与数据集**：使用了 Meta 发布的 **Llama 3.2 (1B 和 3B)** 模型。
*   **测试场景**：模拟端侧推理，涵盖不同的输入序列长度（Input SL）和输出序列长度（Output SL），范围从 64 到 192。
*   **Benchmark 与对比方法**：
    *   **WD (Weight Duplication)**：权重复制方案（基准，内存占用最高）。
    *   **FACIL-O (Oracle)**：假设硬件已解决布局冲突的理想方案。
    *   **Host-only**：不使用 PIM 的纯 CPU 执行方案。
    *   **S-DDB / S-OWR**：本文提出的两种软件变体。

### 4. 资源与算力
*   **硬件平台**：**Samsung Galaxy S24+** 智能手机。
*   **处理器**：Exynos 2400（包含 1x Cortex-X4, 5x A720, 4x A520）。
*   **内存**：LPDDR5X-8533 (68.264 GB/s)。
*   **算力细节**：使用 4 个核心执行 GEMM，2 个核心执行 SMC 拷贝线程。
*   **仿真环境**：由于商用 LPDDR-PIM 硬件尚未完全普及，作者使用了经过工业界验证的 **PIMLibrary** 和 **PIM Simulator** 进行性能仿真和功能验证。

### 5. 实验数量与充分性
*   **实验规模**：作者针对 1B 和 3B 两个模型规模，分别在不同 SL 组合下进行了详尽的测试。
*   **充分性**：实验不仅测量了**首字延迟（TTFT）**和**端到端加速比**，还量化了**内存节省比例**。
*   **客观性**：作者诚实地指出了在短序列下软件重排的开销（无法完全掩盖延迟），并对比了理想硬件方案（FACIL-O），证明了在长序列下软件方案能达到接近硬件方案的性能。

### 6. 主要结论与发现
*   **内存节省**：相比权重复制（WD），PIM-SHERPA 在 Llama 3.2 1B 上节省了 **47.8%** 的内存，在 3B 模型上节省了 **49.7%**。
*   **性能表现**：当输入序列长度 $\ge 128$ 时，DDB 模式可以完全掩盖权重重排的延迟，其 TTFT 性能与理想硬件方案持平。
*   **加速比**：在 LPDDR5X-PIM 环境下，相比纯 CPU 系统，实现了最高 **3.3倍** 的端到端推理加速。
*   **可行性**：证明了在不改变现有 SoC 架构的情况下，通过软件流水线优化即可解决 PIM 部署的核心障碍。

### 7. 优点
*   **高实用性**：纯软件方案，兼容现有的 ExecuTorch 等推理框架，无需等待下一代 SoC 硬件改动。
*   **内存效率极高**：解决了端侧设备运行大模型最核心的内存容量痛点。
*   **首创性**：首次系统性地识别并解决了 PIM 系统中“内存属性不一致”这一隐蔽但关键的问题。

### 8. 不足与局限
*   **短序列开销**：在输入序列较短（如 < 32）时，GEMM 计算时间不足以掩盖 SMC 拷贝时间，会导致 TTFT 明显增加。
*   **调度依赖**：DDB 模式依赖于多核 CPU 的高效调度，如果系统负载过高导致拷贝线程延迟，会产生流水线气泡。
*   **序列长度限制**：实验中测试的最大序列长度为 192，虽然符合目前端侧 RAG 等场景，但对于超长文本（如 128K）的性能表现尚未验证（尽管理论上长文本对该方法更有利）。
