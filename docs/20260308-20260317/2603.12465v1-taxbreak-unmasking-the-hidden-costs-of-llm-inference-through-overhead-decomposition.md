## 摘要
大语言模型（LLM）推理广泛应用于交互式助手和智能体系统。在延迟敏感的部署中，推理时间可能由主机端开销主导。现有方法通常仅将此成本显示为聚合残差或启动/队列指标，这往往不足以识别应优化哪个执行层。本研究提出了 TaxBreak，这是一种基于追踪（trace-driven）的方法论，用于将主机可见的编排开销分解为三个部分：框架转换时间、CUDA 库转换时间以及算子（kernel）启动路径时间。我们在 NVIDIA H100 和 H200 系统上验证了 TaxBreak，并利用它推导出我们提出的主机-设备平衡指数（HDBI），这是一种将设备活动执行与主机可见编排相关联的受限性总结指数。在预填充（prefill）和解码（decode）阶段的代表性稠密（dense）和混合专家（MoE）工作负载中，我们表明仅凭聚合延迟、GPU 空闲时间或受限比率可能会掩盖主要的优化目标。TaxBreak 能够区分优化应减少软件栈开销的情况，以及主要收益来自减少设备端工作量的情况。我们进一步表明，MoE 模型每个输出 token 调度的算子数量比稠密模型多 8-11 倍，对于此类受主机限制的工作负载，CPU 单线程性能是一个一阶参数：更快的宿主机 CPU 可将编排开销降低 10-29%，并将端到端延迟提高多达 14%，即使搭配时钟频率较低的 GPU 也是如此。这些结果使 TaxBreak 成为一种诊断工具，用于评估优化工作应针对软件栈还是设备端的工作负载执行。

## Abstract
Large Language Model (LLM) inference is widely used in interactive assistants and agentic systems. In latency-sensitive deployments, inference time can become dominated by host-side overheads. Existing approaches typically expose this cost only as an aggregate residual or a launch/queue metric, which is often insufficient to identify which execution layer should be optimized. This work presents TaxBreak, a trace-driven methodology for decomposing host-visible orchestration overhead into three components: framework translation time, CUDA library translation time, and kernel launch-path time. We validate TaxBreak on NVIDIA H100 and H200 systems and use it to derive our proposed Host-Device Balance Index (HDBI), a boundedness summary index that relates device-active execution to host-visible orchestration. Across representative dense and mixture-of-experts workloads in both prefill and decode, we show that aggregate latency, GPU inactivity, or boundedness ratios alone can obscure the dominant optimization target. TaxBreak instead distinguishes cases where optimization should reduce software-stack overhead from cases where the primary win comes from reducing device-side work. We further show that MoE models dispatch 8-11x more kernels per output token than dense models, and that for such host-bound workloads, CPU single-thread performance is a first-order parameter: a faster host CPU reduces orchestration overhead by 10-29% and improves end-to-end latency by up to 14%, even when paired with a slower-clocked GPU. These results position TaxBreak as a diagnostic tool for assessing whether optimization effort should target the software stack or the device-side workload execution.

随着大语言模型（LLM）推理（尤其是智能体 Agent 场景）对延迟的要求日益苛刻，**主机端（CPU）的编排开销（Orchestration Overhead）**逐渐成为性能瓶颈。现有的性能分析工具（如 PyTorch Profiler）通常只能提供粗略的聚合数据，将主机开销视为一个“黑盒”残差，难以区分瓶颈究竟源于深度学习框架（如 PyTorch）、供应商库（如 cuBLAS）还是底层的内核启动路径。

本文的研究动机在于：对于像混合专家模型（MoE）这样会产生大量细碎内核调用的工作负载，开发者需要一种细粒度的诊断工具，来决定优化重心应该是软件栈（编译优化）还是硬件执行（增加算力）。

### 2. 论文提出的方法论：TaxBreak
TaxBreak 是一种基于追踪（Trace-driven）的分解方法，其核心思想是将主机侧的总开销 $T_{Orchestration}$ 分解为三个互斥且详尽的部分：

*   **核心公式**：$T_{Host} = \Delta FT + \Delta CT + \Delta KT$
    1.  **$\Delta FT$ (Framework Translation)**：框架转换开销。包含 Python 调度开销和不可缩减的 ATen 调度成本。
    2.  **$\Delta CT$ (CUDA Library Translation)**：CUDA 库转换开销。指调用供应商库（如 cuBLAS/cuDNN）时，在进入内核启动 API 之前的启发式选择、描述符设置等额外成本。
    3.  **$\Delta KT$ (Kernel Launch-path)**：内核启动路径开销。即从调用 `cudaLaunchKernel` 到 GPU 真正开始执行内核的硬件/驱动底层耗时。
*   **HDBI (Host-Device Balance Index)**：主机-设备平衡指数。计算公式为 $T_{DeviceActive} / (T_{DeviceActive} + T_{Orchestration})$。
    *   HDBI 趋近于 0 表示完全受主机限制（Host-bound）；
    *   HDBI 趋近于 1 表示完全受设备限制（Device-bound）。
*   **两阶段测量流程**：
    *   **阶段 1**：全模型追踪，捕获 Python 算子、ATen 算子和内核的关联 ID。
    *   **阶段 2**：隔离重放，通过测量“空内核（Null-kernel）”确定系统底噪，并重放每个唯一内核以精确剥离各层耗时。

### 6. 论文的主要结论与发现
1.  **MoE 的内核爆炸**：在解码阶段，MoE 模型每个 token 调度的内核数量比同规模稠密模型多 **8-11 倍**。这种碎片化导致 MoE 即使在大 Batch Size 下也难以摆脱主机受限（Host-bound）状态。
2.  **CPU 单线程性能是关键**：在主机受限的场景下，CPU 单线程性能是一阶参数。H200 系统虽然 GPU 频率略低，但凭借更快的 CPU，其编排开销降低了 **10-29%**，端到端延迟提升了高达 **14%**。
3.  **HDBI 的诊断价值**：HDBI 能揭示优化手段的本质。例如，FlashAttention-2 虽然降低了主机开销，但它对设备端计算量的削减更剧烈，导致 HDBI 反而下降，这证明其主要贡献在于设备端优化。
4.  **固定成本**：对于稠密模型，主机编排开销在 Eager 模式下相对固定，随着工作负载增加（BS/SL 增大），系统会从主机受限自然转向设备受限。

### 7. 优点
*   **粒度精细**：首次将“框架税”细化到库转换和启动路径级别，填补了性能分析工具的空白。
*   **实用性强**：提出的 HDBI 指数和 TaxBreak 分解法可以直接指导工业界进行推理加速（如决定是否采用 CUDA Graphs 或内核融合）。
*   **视角独特**：通过跨代 CPU/GPU 组合的对比，有力地证明了在现代 LLM 推理中，CPU 性能（尤其是单核性能）被严重低估了。

### 8. 不足与局限
*   **平台依赖**：目前仅针对 NVIDIA GPU 和 CUDA 环境开发，未涵盖 AMD (ROCm) 或其他 AI 加速器。
*   **动态性挑战**：对于高度动态、自动调优或存在复杂同步逻辑的内核，重放机制可能无法 100% 还原真实执行环境中的竞争状态。
*   **软件模式限制**：主要针对 PyTorch Eager 模式。虽然这是目前最灵活的模式，但对于完全编译优化（如全图捕获的 TensorRT）的场景，TaxBreak 的边际效用可能会降低。
