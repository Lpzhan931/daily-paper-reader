## 摘要
推理模型和智能体系统的兴起使得大语言模型的令牌生成延迟成为关键瓶颈。与聊天机器人不同（其延迟增益在人类阅读速度下趋于饱和），这些系统会生成中间推理令牌，这些令牌并非由人类消费。因此，每个令牌的延迟直接决定了端到端的响应时间。低延迟推理采用最小批处理，使得令牌生成受带宽限制。张量并行通过将模型权重分片到多个GPU并并行加载来解决这一问题。然而，扩展到更多GPU会引入随GPU数量增长的全规约开销。移除全规约可将Llama-3.1-8B在8块H200 GPU上的令牌吞吐量提升43%。我们提出了无同步全规约（SiFAR），以减少低延迟推理期间的同步开销。现有的单步和两步算法在通信前后的屏障处引入开销。首先，我们发现单步算法中的底部屏障强制了写后写依赖，并通过协同设计通信和模型执行以启用双缓冲来消除它。然而，单步算法在GPU数量增加时扩展性较差。两步算法在更高TP度数下表现更好，但引入了不可避免的底部屏障。为了克服这一点，我们利用现代交换机中的交换机内规约。我们提出了冗余拉取，其中每个GPU在交换机处规约完整的全规约负载。这提高了单步算法的可扩展性，同时保留了其无底部屏障的优势。最后，为了减少顶部屏障开销，我们观察到每个解码步骤会发起多个全规约操作，使得在第一次之后GPU保持紧密同步。因此，我们提出了推测性规约，它在顶部屏障之前启动数据传输，并通过轻量级验证确保正确性。SiFAR将全规约延迟降低高达52%，并在TP=8时将Llama-3.1-8B的端到端吞吐量提升18.6%，将Qwen3.5-397B-17B的端到端吞吐量提升13.1%。

## Abstract
The rise of reasoning models and agentic systems has made LLM token-generation latency a key bottleneck. Unlike chatbots, whose latency gains saturate at human reading speed, these systems generate intermediate reasoning tokens not consumed by humans. Thus, per-token latency directly determines end-to-end response time. Low-latency inference uses minimal batching, making token generation bandwidth-bound. Tensor Parallelism addresses this by sharding model weights across GPUs and loading them in parallel. However, scaling to more GPUs introduces All-Reduce overheads that grow with GPU count. Removing All-Reduce improves token throughput by 43% for Llama-3.1-8B on 8 H200 GPUs.   We propose Synchronization-Free All-Reduce (SiFAR), which reduces synchronization overhead during low-latency inference. Existing oneshot and twoshot algorithms incur overheads from barriers before and after communication. First, we find that the bottom barrier in oneshot enforces a WAW dependency and eliminate it by co-designing communication and model execution to enable dual buffering. However, oneshot scales poorly with GPU count. Twoshot performs better at higher TP degrees but incurs an unavoidable bottom barrier. To overcome this, we leverage in-switch reduction in modern switches. We propose redundant pull, where each GPU reduces the full All-Reduce payload at the switch. This improves oneshot scalability while retaining its no-bottom-barrier advantage. Finally, to reduce top-barrier overhead, we observe that each decode step issues multiple All-Reduce operations, keeping GPUs tightly synchronized after the first. We therefore propose speculative reduction, which initiates data transfer before the top barrier and ensures correctness via lightweight validation. SiFAR reduces All-Reduce latency by up to 52% and improves end-to-end throughput by 18.6% for Llama-3.1-8B and 13.1% for Qwen3.5-397B-17B at TP=8.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：随着推理模型（如DeepSeek-R1）和智能体系统的兴起，LLM的令牌生成延迟成为关键瓶颈。与聊天机器人不同，这些系统生成的中间推理令牌并非由人类消费，因此每个令牌的延迟直接决定端到端响应时间。降低延迟要求采用最小批处理（batch size = 1），这使得令牌生成变为内存带宽受限。张量并行（Tensor Parallelism, TP）通过将模型权重分片到多个GPU并行加载来提升聚合带宽，从而降低延迟。
- **核心问题**：扩展TP到更多GPU时，全规约（All-Reduce）操作中的同步屏障（barrier）开销成为主要瓶颈。实验表明，在8×H200上运行Llama-3.1-8B，移除All-Reduce可将吞吐量提升43%，而All-Reduce占总时间的30%。传统All-Reduce算法（单步oneshot和两步twoshot）在低延迟推理场景（小负载、小批量）中，同步开销占32–62%的延迟。现有优化如微批处理（microbatching）和细粒度计算-通信重叠需要足够计算量来隐藏通信，但在小批量下无效。
- **整体含义**：本文旨在直接优化All-Reduce本身，减少其同步开销，从而实现低延迟推理的可扩展性。

## 2. 论文提出的方法论

### 2.1 核心思想
通过三种互补技术协同消除All-Reduce中的同步屏障开销：
- **冗余拉取（Redundant Pull）**：利用现代NVIDIA交换机中的`ld_reduce`指令，每个GPU对整个负载（K字节）发起交换机内规约，而不是像oneshot那样从所有对端拉取完整数据（K×(N-1)字节）。这使得数据接收量从K×(N-1)降至K，同时保留单次往返的优势。
- **双缓冲（Dual Buffering）**：消除底部屏障。传统oneshot中底部屏障防止写后写（WAW）依赖——当前GPU在覆盖输入缓冲区前需确保所有对端已完成读取。SiFAR通过分配两个输入缓冲区并在连续All-Reduce操作间交替使用，使缓冲区直到下一次All-Reduce完成后才被重用，从而安全移除底部屏障。实现上通过修改All-Reduce函数签名（增加一个额外参数引用上一缓冲区）迫使内存管理器保持缓冲区存活，无需修改框架内部内存管理。
- **推测性规约（Speculative Reduction）**：减少顶部屏障开销。顶部屏障确保所有GPU已产生有效部分结果，其开销主要来自标志交换（flag exchange）而非GPU间分歧（因为CUDA Graph或Megakernel使GPU在后继操作中保持紧密同步）。SiFAR让每个GPU在进入All-Reduce时**推测性**地立即开始从对端获取数据，同时每个线程块写一个标志到验证缓冲区。该标志随负载一起规约；若规约结果等于N×flag，则推测成功；否则重试。验证逻辑依赖GPU单调性假设：若GPU1在T1时刻领先GPU0，则在T2>T1时仍领先（因CUDA Graph消除主机端可变性且GPU内核以运行至完成模型执行）。实验证实无静默错误。

### 2.2 关键技术细节
- **冗余拉取实现**：用`multimem.ld_reduce(payload_buf)`替代oneshot中对每个对端的循环拉取与本地归约。
- **双缓冲实现**：在Megakernel中，将All-Reduce函数签名改为`redundant_pull(input_buf1, input_buf0)`，其中`input_buf0`是前一次All-Reduce的输入缓冲区。这强制内存管理器在第二次All-Reduce执行前保留`input_buf0`，避免复用。
- **推测性规0实现**：每个线程块用一个额外线程束处理验证缓冲区（4字节标志），其余线程束处理负载。验证缓冲区采用本地拉取-归约（`local_reduce`），而非交换机规约，因前者经验上更快。若验证失败，使用`.cg`修饰符绕过L1缓存加载最新标志，并重试负载规约。每次All-Reduce后增加标志值，确保唯一验证结果。双缓冲也应用于验证缓冲区以防止竞争。
- **Megakernel集成**：在Megakernel中，复用已有操作符完成标志作为验证输入，无需额外写入验证缓冲区。

## 6. 主要结论与发现

- **SiFAR大幅降低All-Reduce延迟**：最高52%（TP=8, 8KB负载下从5.11μs降至2.44μs）。
- **端到端吞吐量提升显著**：Llama-3.1-8B在TP=8时提升18.6%；Qwen3.5-397B-17B在TP=8时提升13.1%。
- **三项技术均有贡献**：在TP=8下，冗余拉取贡献6.7%，双缓冲额外贡献3.4%，推测性规约贡献8.5%。越高的TP度数，优化效果越明显。
- **误推测开销极小**：尽管平均误推测率达32.2%（Llama-3.1-8B, TP=8），但仅导致1.6%的性能损失；验证开销损失1.4%。
- **尾部延迟无恶化**：p99延迟降至基线的84.0%（Llama）和87.8%（Qwen），说明重试虽存在但不影响尾部。
- **计算-通信融合无效**：在小批量下，融合反而更慢，SiFAR直接降低通信延迟优于隐藏。
- **与现有方案比较**：优于TRT-LLM、Lamport、NCCL 2.30。

## 7. 优点

1. **问题定位精准**：识别出低延迟推理场景下同步屏障是All-Reduce的主要开销（32-62%），并直接针对屏障优化。
2. **创新性技术组合**：
   - 冗余拉取巧妙扩展交换机规约的用途，从reduce-scatter变为完整负载规约，兼具twoshot的低数据传输和oneshot的单次往返优势。
   - 双缓冲通过最低限度的代码改动（仅修改函数签名）实现，无需侵入框架内存管理，实用性强。
   - 推测性规约利用GPU同步特性消除标志交换，验证机制轻量且正确性有理论保障（单调性假设）。
3. **实验严谨**：消融实验量化每个部分的增益和开销；对比多种基线（包括NCCL最新版本）；验证尾部延迟未受重试影响；在两种不同架构模型上评估。
4. **开源友好**：基于Megakernel实现，代码可复现；无专有硬件依赖（仅需NVSwitch支持的`ld_reduce`，目前已在H100/H200可用）。
5. **实际价值**：针对当前LLM部署中最关键的延迟瓶颈，直接提升小批量下解码吞吐，契合推理模型和智能体的需求。

## 8. 不足与局限

1. **硬件依赖性**：依赖NVIDIA NVSwitch的`ld_reduce`指令，不适用于无交换机内规约的集群（如InfiniBand RDMA）。未来多节点场景下可能无法直接迁移。
2. **可扩展性有限**：仅在单节点内测试（TP≤8）。随着GPU数量进一步增加（如TP=16、32），交换机内的规约负载可能增长，冗余拉取的延迟可能增加。论文未提供更大规模数据。
3. **未考虑多节点部署**：实际生产可能需跨节点张量并行，SiFAR尚未处理节点间通信同步。
4. **单调性假设的潜在风险**：虽然论文通过实验验证了100,000次迭代无静默错误，但极端情况下（如GPU频率变化、内存错误）可能破坏单调性，导致验证失败而未被检测。论文未讨论容错机制。
5. **仅适用decode阶段**：预填充阶段（prefill）为计算密集型，All-Reduce开销占比小，SiFAR收益有限。论文未测试prefill。
6. **模型范围有限**：仅测试两个模型（8B和MoE-397B），未覆盖更大规模密集模型（如Llama-3.1-70B、405B）或不同架构（如Mamba、混合专家变体）。
7. **能耗评估缺失**：未报告SiFAR相比基线是否增加功耗或能耗效率如何。
8. **与其他优化的互补性未探索**：如程序化依赖启动（PDL）、Mirage自动融合等，SiFAR声称正交但未联合实验量化整体收益。
