## 摘要
推测解码通过将草稿生成与目标验证解耦来加速大语言模型推理。最近的并行草稿模型能够在单次前向传播中高效生成长令牌序列，但由于缺乏令牌间依赖关系，其接受率会迅速下降。此外，不加区分地验证这些扩展块会将宝贵的批处理容量浪费在高拒绝风险的令牌上，严重降低高并发服务系统的吞吐量。我们提出DSpark，一个将高吞吐并行生成与自适应、负载感知验证相统一的推测解码框架。为保持草稿质量，DSpark采用半自回归架构，将并行骨干与轻量级顺序模块耦合，引入块内依赖建模并缓解后缀衰减。为优化系统效率，DSpark采用基于置信度调度的验证，根据估计的前缀生存概率和引擎特定的吞吐量特征，动态调整每个请求的验证长度。在跨不同领域的离线基准测试中，DSpark相较于最先进的自回归和并行草稿模型显著提高了接受长度。当在DeepSeek-V4服务系统中部署于实时用户流量下时，DSpark成功减轻了验证浪费。与已建立的生产基线（MTP-1）相比，DSpark在匹配吞吐水平下将每用户生成速度提升了60%到85%。更重要的是，通过防止在严格交互性约束下吞吐量的严重下降，它实现了以前无法达到的性能等级，推动了服务系统的帕累托前沿。

## Abstract
Speculative decoding accelerates Large Language Model (LLM) inference by decoupling draft generation from target verification. While recent parallel drafters efficiently propose long token sequences in a single forward pass, they suffer from rapid acceptance decay due to a lack of inter-token dependencies. Furthermore, indiscriminately verifying these extended blocks wastes critical batch capacity on tokens with high rejection risks, severely degrading throughput in high-concurrency serving systems. We introduce DSpark, a speculative decoding framework that unifies high-throughput parallel generation with adaptive, load-aware verification. To maintain draft quality, DSpark utilizes a semi-autoregressive architecture, coupling a parallel backbone with a lightweight sequential module, to introduce intra-block dependency modeling and mitigate suffix decay. To optimize system efficiency, DSpark employs confidence-scheduled verification, dynamically tailoring the verification length for each request based on estimated prefix survival probabilities and engine-specific throughput profiles. On offline benchmarks across diverse domains, DSpark substantially improves the accepted length over state-of-the-art autoregressive and parallel drafters. When deployed within the DeepSeek-V4 serving system under live user traffic, DSpark successfully mitigates verification waste. Compared to the established production baseline (MTP-1), DSpark accelerates per-user generation speeds by 60 to 85 percent at matched throughput levels. More importantly, by preventing severe throughput degradation under strict interactivity constraints, it enables performance tiers that were previously unattainable, shifting the Pareto frontier of our serving system.

# DSpark 论文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：大规模语言模型（LLM）自回归推理延迟高，推测解码通过轻量草稿模型生成候选令牌、再由目标模型并行验证来加速。现有并行草稿模型（如DFlash）可在单次前向传播中生成大量草稿令牌，但各位置独立预测，缺乏令牌间依赖，导致**接受率随位置快速衰减（后缀衰减）**。同时，**无差别地验证完整草稿块会浪费批处理容量**：在高并发生产系统中，低置信度的尾部令牌占用宝贵资源，降低整体吞吐量。
- **目标**：设计一个能同时提升草稿质量（通过引入块内依赖）并动态适应系统负载的推测解码框架，以在保持无质量损失的前提下最大化端到端加速。

## 2. 方法论：核心思想与技术细节

### 2.1 整体架构
DSpark 包含两个互补组件：
1. **半自回归生成（Semi-Autoregressive Generation）**：采用“并行骨干 + 轻量级顺序头”结构。
2. **置信度调度验证（Confidence-Scheduled Verification）**：结合置信度头与硬件感知调度器，动态决定每个请求的验证长度。

### 2.2 半自回归生成
- **并行骨干**：采用 DFlash 作为基础，稍作修改（将锚点 token 也作为第一个预测位置）。单次前向传播输出所有位置的隐藏状态和基础 logits \(U_1,\dots,U_\gamma\)。
- **顺序模块**：在基础 logits 上添加依赖当前前缀的 bias \(B_k\)，产生条件分布：
  \[
  p_k(v|x_0,x_{<k}) = \frac{\exp(U_k(v) + B_k(x_0,x_{<k},v))}{\sum_u \exp(U_k(u) + B_k(x_0,x_{<k},u))}
  \]
  两种实现：
  - **Markov 头**：仅依赖前一个 token，采用低秩分解 \(B = W_1 W_2\)（\(r=256\)）。
  - **RNN 头**：通过隐状态 \(s_k\) 累积整个前缀信息，采用门控更新。
- **优势**：并行骨干保持草稿延迟 \(T_{\text{draft}}\) 几乎与块大小 \(\gamma\) 无关；顺序头极小增加延迟，但显著提升后缀接受率。

### 2.3 置信度调度验证
#### 2.3.1 置信度头
- 为每个草稿位置输出一个标量 \(c_k \in (0,1)\)，表示**给定之前所有 token 已被接受条件下该位置生存的条件概率**。
- 架构：线性投影 + sigmoid，输入为骨干隐藏状态和 Markov Embedding。
- 监督信号：理论接受率 \(c_k^* = 1 - \frac{1}{2}\|p_k^d - p_k^t\|_1\)。
- **后处理校准（Sequential Temperature Scaling, STS）**：由于原始置信度过度自信，在验证集上从左到右对每个位置独立搜索最佳温度，最小化预期校准误差（ECE），得到校准后的累积概率。

#### 2.3.2 硬件感知前缀调度器
- **问题**：静态阈值在多请求生产环境中次优，因为验证额外 token 的收益与当前系统负载相关。
- **算法**（Algorithm 1）：
  1. 对每个请求 \(r\)，计算前缀生存概率 \(a_{r,j} = \prod_{i \le j} c_{r,i}\)。
  2. 将所有候选项 \((r,j)\) 按 \(a_{r,j}\) 降序排序。
  3. 贪心地依次采纳候选项，每次计算当前吞吐量 \(\Theta = \tau^* \cdot \text{SPS}(B)\)，其中 \(\tau^*\) 是期望接受数，\(B\) 是批大小，\(\text{SPS}(B)\) 是预置的性能曲线。
  4. 当吞吐量下降 (\(\Theta \le \Theta_{\text{best}}\)) 时**提前停止**，避免未来 token 信息的泄漏，保证无损性。
- **实际部署调整**：使用异步调度，基于两步之前的置信度近似确定截断长度，兼容 CUDA graph replay 和零开销调度（ZOS）。

### 2.4 训练
- **数据**：从目标序列中随机采样多个锚点位置，形成 \(\gamma\)-token 块。
- **冻结目标模型**，仅更新草稿骨干、顺序块和置信度头。
- **损失函数**：
  \[
  \mathcal{L} = \alpha_{\text{ce}} \mathcal{L}_{\text{ce}} + \alpha_{\text{tv}} \mathcal{L}_{\text{tv}} + \alpha_{\text{conf}} \mathcal{L}_{\text{conf}}
  \]
  - \(\mathcal{L}_{\text{ce}}\)：交叉熵，训练草稿预测 ground-truth token。
  - \(\mathcal{L}_{\text{tv}}\)：分布匹配损失，最小化草稿与目标分布的总变差距离，直接最大化接受率。
  - \(\mathcal{L}_{\text{conf}}\)：二元交叉熵，训练置信度头预测软标签 \(c_k^*\)。
  - 所有损失项按位置加权 \(w_k = \exp(-(k-1)/\gamma)\)，强调较早位置。

## 6. 论文的主要结论与发现

1. **半自回归生成有效缓解后缀衰减**：DSpark 结合并行骨干的高初始接受率和顺序头的依赖建模，在离线基准上平均接受长度显著优于 Eagle3（+26.7%~30.9%）和 DFlash（+16.3%~18.4%）。
2. **置信度调度验证提升系统效率**：通过校准的置信度头和硬件感知前缀调度，动态调整验证长度，在高并发下避免了验证浪费，同时在低并发下充分利用空闲算力。
3. **生产环境中实现帕累托前沿移动**：在 DeepSeek-V4 部署中，DSpark 在匹配吞吐能力下使每用户生成速度提升 60%~85%（V4-Flash）和 57%~78%（V4-Pro）。在严格交互要求下，防止了吞吐量悬崖式下降。
4. **架构分析揭示关键 trade-off**：并行草稿第一位置容量优势远超自回归草稿，但后缀衰减严重；DSpark 的轻量顺序模块以极低延迟代价弥补了这一缺陷（Figure 2, 4）。

## 7. 优点

- **方法创新**：首次将半自回归生成与全局吞吐量调度有机结合，同时解决了草稿质量和系统效率两大瓶颈。
- **实践落地**：提出的硬件感知调度器支持生产级部署（异步、CUDA graph 兼容），并开源了模型和训练仓库（DeepSpec），促进社区研究。
- **实验完整性**：从离线消融到在线生产验证，覆盖多模型、多域、多负载场景，并提供位置级分析解释机制。
- **理论保障**：调度算法通过早期停止保证无损性，校准方法（STS）确保了概率的可靠性。

## 8. 不足与局限

1. **固定草稿生成开销**：即使能被调度器提前截断，并行骨干仍需为所有 \(\gamma\) 个位置执行一次前向传播。对于本质上低接受率的请求（如高熵对话），这部分计算仍无法回收。
2. **环境依赖**：性能曲线 \(\text{SPS}(B)\) 需预先 profiling，实际系统可能因异构硬件或动态负载而波动，未评估其鲁棒性。
3. **超参数未详细讨论**：训练损失权重（\(\alpha_{\text{ce}}=0.1, \alpha_{\text{tv}}=0.9, \alpha_{\text{conf}}=1.0\)）和位置加权指数仅给出默认值，未进行消融研究。
