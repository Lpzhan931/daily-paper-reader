## 摘要
LLM智能体正成为研究、编程和问答的常见界面，但其思考-行动-观察循环通常是串行的：模型推理，发出工具调用，然后闲置GPU直到结果返回。在我们的工作负载中，这种等待消耗了16-37%的墙钟时间，在先前报告中为35-61%。推测性工具执行可以隐藏这种等待，但现有系统需要辅助预测器、历史痕迹或静态工作流图，留下了无需训练、即日部署的空白。我们观察到模型可以成为自己的预测器：在生成开始时分叉的一个探针以74.6-99.6%的准确率预测Qwen3-32B即将使用的工具名称，跨五个基准测试。我们提出SPORK（自推测分叉），一个无需训练的控制器，它提前分派推测的工具调用，使其执行与剩余的思维链解码重叠。一个成本模型捕捉推测何时达到盈亏平衡，每个组件改进其一个项：前缀缓存分叉降低探针成本，置信门过滤错误预测，部分令牌接受将拒绝的探针转换为推测解码草稿。在接受时，工具结果在推理结束时准备好；在拒绝时，SPORK回退到串行执行，没有正确性惩罚。在真实工具基准测试中，SPORK将Qwen3-32B的GAIA P95降低18%（从131.9秒到108.1秒）；该机制在从4B到32B的模型大小以及密集和混合专家模型中均有效，任务准确率在基准的1个百分点以内或更好（在测量之处）。SPORK作为标准补全API之上的薄控制器部署（无需重新训练，无需辅助模型，无需离线痕迹），并且与令牌级推测解码正交。SPORK开源在https://github.com/baihuajun24/spork。

## Abstract
LLM agents are becoming a common interface for research, coding, and question answering, yet their Thought-Action-Observation loop is often serial: the model reasons, emits a tool call, then idles the GPU until the result returns. This wait consumes 16-37% of wall time in our workloads and 35-61% in prior reports. Speculative tool execution can hide this wait, but existing systems need auxiliary predictors, historical traces, or static workflow graphs, leaving a gap for training-free, day-one deployment. We observe that the model can be its own predictor: a probe forked at the start of generation predicts Qwen3-32B's upcoming tool name with 74.6-99.6% accuracy across five benchmarks. We present SPORK (Self-sPeculative fORKing), a training-free controller that dispatches the speculated tool call early, overlapping its execution with the remaining chain-of-thought decode. A cost model captures when speculation breaks even, and each component improves one of its terms: a prefix-cache fork cuts probe cost, a confidence gate filters mispredictions, and partial-token accept turns rejected probes into speculative-decoding drafts. On acceptance, the tool result is ready when reasoning ends; on rejection, SPORK falls back to serial execution with no correctness penalty. On real-tool benchmarks, SPORK cuts Qwen3-32B's GAIA P95 by 18% (131.9 to 108.1 s); the mechanism holds across model sizes from 4B to 32B and across dense and mixture-of-experts models, with task accuracy within 1 pp of baseline or better wherever measured. SPORK deploys as a thin controller over standard completion APIs (no retraining, no auxiliary models, no offline traces) and is orthogonal to token-level speculative decoding. SPORK is open source at https://github.com/baihuajun24/spork.

### 1. 核心问题与整体含义（研究动机和背景）

- **问题**：LLM 智能体的典型执行模式是串行的 ReAct 循环（思考→行动→观察）。模型生成工具调用后必须等待外部工具返回结果，GPU 在此期间闲置。论文测量发现，工具等待消耗了 16–37% 的端到端墙钟时间（在先前工作中高达 35–61%）。随着 LLM 推理速度持续提升（例如通过推测解码、硬件扩展），工具等待所占比例反而增加，成为延迟的主要瓶颈。
- **背景**：现有方案（如令牌级推测解码、并行工具调用、工作流优化器、基于辅助预测器或历史痕迹的智能体推测系统）要么无法提前分派工具，要么需要外部训练数据、辅助模型或静态图，无法实现“即日部署、无需训练”的目标。
- **机会**：论文观察到，指令微调模型在生成早期（第一个令牌后）就已“知道”其即将调用的工具名称。Qwen3-32B 在五个基准测试上的工具名称预测准确率达到 74.6–99.6%。利用这一早期意图，可以提前执行工具调用，与后续推理解码并行，从而隐藏等待时间。

### 2. 方法论：核心思想、关键技术细节、算法流程

- **核心思想**：将运行中的 LLM 自身作为其工具调用的预测器，通过分叉（fork）当前生成状态，插入一个强制工具调用前缀的探针，在模型尚未完成完整思维链时预先推测即将发出的工具调用。若推测正确，则工具结果在模型推理结束时已就绪；若错误，则回退到串行执行，保证正确性无损失。
- **关键技术细节**：
  - **D1：前缀缓存分叉（Prefix-Cache Fork）**  
    主线程生成第一个令牌后，分叉线程发起一个强制工具调用前缀的补全请求。由于主线程已填充前缀 KV 缓存，分叉的预填充成本极低（从约 1.3 s 降至约 0.05 s），探针总开销从 1.6 s 降至 0.35 s。
  - **D2：自适应置信门控（Confidence Gate）**  
    探针生成工具调用时记录每令牌的 logprob，计算工具名称跨度上的最小 top-1 概率作为置信度得分。当置信度超过阈值 θ（实验设为 0.90）时，才提交推测性工具执行。该门控在保留所有正确探针的同时过滤了 77% 的错误探针，将接受率 α 从 0.22 提升至 0.37。
  - **D3：部分令牌接受（Partial-Token Accept）**  
    当严格匹配拒绝探针时，将探针的工具调用令牌序列作为草稿（draft）注入主线程的推测解码验证路径。主线程验证并接受最长匹配前缀（中位数 18 个令牌），剩余部分重新解码，从而将拒绝探针的平均浪费从全量解码（约 50 令牌）降低为仅解码后缀（约 20 令牌）。
- **公式（成本模型）**：  
  \(\text{Ratio} = \frac{T_{\text{base}}}{T_{\text{base}}^* - \alpha \cdot t_{\text{overlap}} + T_{\text{oh}}}\)  
  其中 α 为接受率，\(t_{\text{overlap}}\) 为平均重叠时间，\(T_{\text{oh}}\) 为开销。当 \(\alpha \cdot t_{\text{overlap}} \ge T_{\text{oh}}\) 时，推测加速有效。
- **算法流程**（简化）：
  1. 以流式聊天补全请求发起主线程。
  2. 主线程输出第一个令牌后，分叉线程以强制工具调用前缀发起补全。
  3. 分叉线程监控生成 logprobs；当置信度 ≥ θ，提交推测工具执行（与主线程剩余解码并行）。
  4. 主线程完成工具调用后，与提交的探针进行严格比较。若匹配，直接使用预取结果；若不匹配，回退到串行执行，同时 D3 将拒绝的探针令牌作为草稿进行推测解码验证。

### 6. 主要结论与发现

- SPORK 有效隐藏工具等待时间：在 GAIA 上，Qwen3-32B 的 P95 延迟降低 18%（131.9 → 108.1 s）；在 HotpotQA 上加速比 1.06× 且准确率提升 1.5 pp（Pareto 最优）；在 tau2 上加速比随工具延迟单调上升（1.09× 到 1.18×）。
- 机制在模型规模和架构上通用：4B、32B、MoE 均有效，且任务准确率保持不变或更好。
- 自推测优于外部署模型：在可控实验中，自推测的接受率更高，浪费的推测执行更少，且无需额外 GPU。
- 成本模型准确预测了收益区间：工具延迟 ≥ 约 1 s 时收益明显，短工具或非思维链模式下收益消失。
- 总体结论：SPORK 是一种简单、高效、无需训练、可开箱即用的智能体推理加速方法。

### 7. 优点

- **无需训练或收集数据**：完全基于运行中模型自身的状态，适合即日部署。
- **正交于令牌级推测解码**：可叠加使用，进一步提升加速效果。
- **成本模型指导实践**：使用者可通过测量 fork 准确率和工具延迟快速估计收益，决定是否启用。
- **正确性保证**：严格匹配确保推测错误时回退到串行，不会污染对话历史。
- **工程友好**：作为轻量级控制器（Python），通过标准 API 与 vLLM 交互，集成成本低。
- **开源**：代码已公开。

### 8. 不足与局限

- **仅支持只读工具**：对于具有副作用的写操作工具，无法安全回滚，当前不支持。未来需配合沙箱或检查点机制。
- **依赖开放权重模型与引擎 API**：需要使用支持 /v1/completions、per-token logprobs 和前缀缓存共享的服务引擎。闭源 API 无法实现分叉。
- **高并发下收益可能缩小**：探针流量可能与主请求争抢批次槽位，导致延迟收益下降。论文仅提及，未详细量化。
- **需要思维链模式**：非思考模式（no-think）下推理时间短（0.3–0.9 s），重叠窗口太小，推测无效。
- **格式不匹配时完全失效**：如模型原生工具调用格式与强制探针不一致（如 Qwen3.5 的 XML 格式导致名称准确率仅 4.5%），需改为 JSON 提示才能工作。
- **实验覆盖有限**：仅使用 Qwen 系列模型，未在其他主流 LLM（如 LLaMA-3、Gemma、Mixtral）上验证；仅使用单一服务引擎（vLLM），未在 SGLang 等其他引擎上测试。
- **未系统研究吞吐量-延迟权衡**：论文主要关注延迟优化，对于高吞吐批处理场景下的性能分析较为粗略。
