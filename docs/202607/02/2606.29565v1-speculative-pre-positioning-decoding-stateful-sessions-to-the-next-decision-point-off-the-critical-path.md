## 摘要
无状态推理服务器（vLLM、SGLang、TensorRT-LLM）在请求之间空闲，而加速器在等待；有状态会话则利用了这段空闲时间。推测性预定位使用目标模型自身的前向传播且无需草稿模型，将会话解码到其下一个决策点，将跨请求预填充和入口解码移出关键路径：下一个请求从其增量的预付入口恢复，或者当置信门触发时，通过一次近乎恒定的词汇表扫描从缓存分布中应答，无需解码，代价仅为能量消耗和偶尔发生的、有界的错误接受。收益取决于模型能力：有能力的模型以近乎全覆盖和约87%的精确度触发置信门（较小的模型则永远无法触发），返回第一个令牌约需1.0毫秒，而前缀缓存仍需要39毫秒的解码时间。

## Abstract
A stateless inference server (vLLM, SGLang, TensorRT-LLM) idles between requests while the accelerator waits; a stateful session reclaims that idle time. Speculative pre-positioning decodes the session forward to its next decision point with the target model's own forward pass and no draft model, moving the cross-request prefill and entry-decode off the critical path: the next request resumes from a pre-paid entry on its delta, or, when a confidence gate fires, is answered from a cached distribution in one near-constant vocabulary scan with no decode, at a cost only of energy and a rare, bounded false accept. The payoff is conditional on capability: a capable model fires the gate at near-full coverage and about 87% precision (a smaller one never clears it), returning the first token in about 1.0 ms versus the 39 ms decode a prefix cache still pays.

- **问题**：无状态推理服务器在请求之间空闲，加速器等待；前缀缓存虽然重用了 KV 状态，但仍需在关键路径上执行一次解码步骤（约 39 ms）才能产生第一个 token。

### 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：在空闲窗口内运行一次目标模型的前向传播，计算并缓存决策点处的输出分布；下一个请求到来时，若置信门条件满足，则直接返回缓存的 top-1 token（一次词汇表扫描），否则退回到依然比基线便宜的路径。
- **关键技术细节**：
  - **决策点与空闲窗口**：决策点是下一个请求恢复生成的位置（$p = n_s + L_{\text{entry}}$）；空闲窗口 $I_{\text{idle}}$ 是状态就绪到请求到达的时间间隔。
  - **就绪分布缓存**：缓存决策点处的 logits（或仅 top token + gap），而非仅 KV 状态。
  - **置信门**：基于 logit gap $\gamma = \ell_{(1)} - \ell_{(2)}$ 和答案集 $A$，当 $\gamma > \tau$ 且 $\arg\max(\ell) \in A$ 时触发快速路径；阈值 $\tau$ 通过校准选择以控制错误率。
  - **失效规则**：任何改变状态的突变（滑动窗口、回滚、新更新）原子性地清除缓存和有效标志，确保永不返回过时预测。
- **算法流程**：
  - **算法 1（空闲时预定位）**：会话更新后，若 entry 长度在预算内，则追加 entry、运行一次前向传播，记录决策点位置、就绪分布和有效标志。
  - **算法 2（置信门快速路径服务）**：请求到达时，若有效且 entry 匹配，则检查门条件；通过则直接返回缓存 token（成本 $L_{\text{fast}}$），否则退回到正常路径（成本 $L_{\text{fall}}$，已预付 entry 预填充）。

- **能耗 vs 收益**：在查询密集场景下几乎每次预定位都被消耗；在更新密集场景下浪费能量但不影响延迟。
