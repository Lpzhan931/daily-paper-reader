## 摘要
大语言模型代理正成为研究、编程和问答的常见接口，但其思考-行动-观察循环通常为串行：模型进行推理，发出工具调用，然后让GPU空闲直至结果返回。在我们的工作负载中，这种等待消耗了挂钟时间的16-37%，在先前报告中则占35-61%。推测性工具执行可以隐藏这种等待，但现有系统需要辅助预测器、历史记录或静态工作流图，因此存在无需训练、即日部署的空白。我们观察到模型自身可以作为预测器：在生成开始时分支出的探针在五个基准测试上以74.6-99.6%的准确率预测Qwen3-32B即将使用的工具名称。我们提出SPORK（自我推测分支），一种无需训练的控制器，它提前调度推测性工具调用，使其执行与剩余思维链解码重叠。成本模型捕捉推测何时达到盈亏平衡，每个组件改进其一项项：前缀缓存分支降低探针成本，置信门控过滤错误预测，部分令牌接受将拒绝的探针转化为推测解码草稿。若接受，则推理结束时工具结果已就绪；若拒绝，SPORK回退到串行执行且无正确性惩罚。在真实工具基准测试上，SPORK将Qwen3-32B的GAIA P95降低18%（从131.9秒降至108.1秒）；该机制适用于从4B到32B的模型规模以及密集和混合专家模型，在可测量处任务准确率与基线相差1个百分点以内或更优。SPORK作为薄控制器部署在标准完成API之上（无需重新训练、无需辅助模型、无需离线记录），且与令牌级推测解码正交。SPORK开源地址：https://github.com/baihuajun24/spork。

## Abstract
LLM agents are becoming a common interface for research, coding, and question answering, yet their Thought-Action-Observation loop is often serial: the model reasons, emits a tool call, then idles the GPU until the result returns. This wait consumes 16-37% of wall time in our workloads and 35-61% in prior reports. Speculative tool execution can hide this wait, but existing systems need auxiliary predictors, historical traces, or static workflow graphs, leaving a gap for training-free, day-one deployment. We observe that the model can be its own predictor: a probe forked at the start of generation predicts Qwen3-32B's upcoming tool name with 74.6-99.6% accuracy across five benchmarks. We present SPORK (Self-sPeculative fORKing), a training-free controller that dispatches the speculated tool call early, overlapping its execution with the remaining chain-of-thought decode. A cost model captures when speculation breaks even, and each component improves one of its terms: a prefix-cache fork cuts probe cost, a confidence gate filters mispredictions, and partial-token accept turns rejected probes into speculative-decoding drafts. On acceptance, the tool result is ready when reasoning ends; on rejection, SPORK falls back to serial execution with no correctness penalty. On real-tool benchmarks, SPORK cuts Qwen3-32B's GAIA P95 by 18% (131.9 to 108.1 s); the mechanism holds across model sizes from 4B to 32B and across dense and mixture-of-experts models, with task accuracy within 1 pp of baseline or better wherever measured. SPORK deploys as a thin controller over standard completion APIs (no retraining, no auxiliary models, no offline traces) and is orthogonal to token-level speculative decoding. SPORK is open source at https://github.com/baihuajun24/spork.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：LLM 代理的 ReAct 循环（思考-行动-观察）本质上是串行的：模型先推理（Chain-of-Thought, CoT），然后发出一段工具调用，在工具执行期间 GPU 空闲等待，直到结果返回才能继续生成。这种“工具等待”占端到端墙钟时间的 16–37%（本文测量值），在先前文献中甚至高达 35–61%。
- **背景**：随着 LLM 推理速度的不断提升（如推测解码、硬件扩展、分离式服务），工具等待时间占比反而会增加，成为代理延迟的渐近瓶颈。已有一些推测执行方案（如 Speculative Actions、DualSpec、PASTE）试图通过提前执行工具调用隐藏等待，但它们依赖辅助预测模型、历史轨迹或静态工作流图，不能做到“即开即用”（训练无关、无需离线数据）。
- **核心含义**：本文提出一种**无训练、自推测**的方法，利用正在运行的 LLM 本身作为预测器，在生成初期就推测出即将调用的工具并提前执行，从而将工具执行与剩余推理解码重叠，显著降低端到端延迟且不影响正确性。

## 2. 方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- **自推测分叉（Self-Speculative Forking）**：在 LLM 生成主线程的第一个 token 后，分出一个探针（probe）子线程，强制以工具调用前缀开始生成，预测主线程最终会调用的工具名称和参数。
- 三个实验洞察：
  - **洞察1**：模型在推理早期就已“知道”自己将调用哪个工具。对 Qwen3-32B，在 fork 点预测工具名称的准确率达 74.6–99.6%（不同基准），但参数准确率较低（7.6–83.9%）。
  - **洞察2**：探针的 log-probability 置信度可以区分正确和错误预测，且随着观察到的 CoT 长度增加，参数准确率从 7.6% 升到 97.5%（接近完整 CoT）。这意味着可以动态等待直到置信度足够高才执行投机调度。
  - **洞察3**：即使严格匹配拒绝探针，被拒绝的探针与最终工具调用之间仍有中位数 18 个共享 token（占工具调用体的 35–50%），这些 token 可作为推测解码的草稿被主线程验证重用。

### 关键技术细节（三个设计 D1–D3）

| 设计 | 对应洞察 | 成本模型效应 | 实现要点 |
|------|----------|--------------|----------|
| **D1：前缀缓存分支** | 洞察1 | 最小化探针开销 \(T_{\text{oh}}\) | 在主线程第一个 streaming token 后分叉，利用 vLLM 的 prefix cache 命中，将探针预填充时间从 1.3s 降到 0.05s，总探针开销从 1.6s 降到 0.35s。 |
| **D2：自适应置信门控** | 洞察2 | 最大化期望重叠 \(\alpha \cdot t_{\text{overlap}}\) | 探针生成时监测工具名称 span 的 log-probability，使用最小值作为置信度 \(c\)。阈值 \(\theta=0.90\)（按 F1 最大选择）。只有当 \(c \ge \theta\) 时才提交投机工具执行，否则等待更多 CoT 或回退。 |
| **D3：部分 token 接受** | 洞察3 | 降低拒绝分支的有效解码成本 | 将拒绝探针的工具调用体作为推测解码的草稿，主线程在工具调用边界处验证并与贪婪解码匹配，接受最长共同前缀，只解码分歧后的剩余内容。平均节省约 0.6s 解码时间/拒绝轮次。 |

### 成本模型
- **公式**：\( \text{Ratio} = T_{\text{base}} / (T_{\text{base}}^* - \alpha \cdot t_{\text{overlap}} + T_{\text{oh}}) \)
  - \(T_{\text{base}}\)：基线串行时间（解码+工具执行）
  - \(T_{\text{base}}^*\)：投机路径上的主线程解码成本（D3 降低后）
  - \(\alpha\)：门控接受率（按轮次计）
  - \(t_{\text{overlap}}\)：每接受轮次中工具执行与解码重叠的平均时间
  - \(T_{\text{oh}}\)：探针计算和拒绝轮次浪费的开销
- **盈亏平衡条件**：\( \alpha \cdot t_{\text{overlap}} \ge T_{\text{oh}} \)（近似形式）

### 算法流程（Algorithm 1）
1. 派发主线程 streaming chat completions。
2. 等主线程第一个 streaming token 后，开始周期性地分叉探针：使用强制前缀生成工具调用，计算置信度。
3. 当置信度达到阈值，立即异步执行推测工具（与主线程解码重叠）。
4. 等主线程完成并生成实际工具调用后，对比探针与实际调用：若严格匹配则使用预计算结果；否则回退到串行执行，同时 D3 将探针 body 作为草稿注入主线程解码以加速。

## 3. 实验设计

## 6. 主要结论与发现

1. **延迟降低显著**：在 GAIA 上，Qwen3-32B 的 P95 延迟从 131.9s 降至 108.1s（-18%）；P50 从 34.7s 降至 31.2s（-10%）。tau2 上随工具延迟增加，加速比从 1.09× 单调升到 1.18×。
2. **正确性保持**：所有基准测试的任务准确度（EM/F1）与基线相差在 1 个百分点以内，有些甚至略有提升（如 HotpotQA 上 +1.5pp EM）。
3. **跨模型泛化**：加速效果在 Qwen3-4B、Qwen3-32B、Qwen3.5-35B-A3B（MoE）上均成立，其中 MoE 模型在 GAIA、tau2、HotpotQA 上 P95 降低 16–20%。
4. **成本模型准确**：tau2 延迟扫描表明，预测加速比与实测加速比偏差最大仅 1.84%，验证了式 (1) 的有效性。
5. **D1+D2+D3 协同最佳**：D1 改善均值/中值，D2+D3 改善尾延迟，三者组合达到最佳的 P95。
6. **自推测优于跨模型推测**：与使用小模型（Qwen3-4B）作为探针对比，自推测接受率更高（0.792 vs 0.590），节省更多工具延迟，且无需额外 GPU 模型。

## 7. 优点

- **无训练、即用部署**：不需要额外训练、辅助模型、历史轨迹或静态图，只需标准 API 支持和前缀缓存。
- **机制简洁而稳健**：利用 LLM 自身的 logit 信号作置信门控，无需复杂分类器；拒绝时通过 D3 回收 token，零额外开销时还能加速主线程。
- **正交性**：与现有的 token 级推测解码（如美杜莎、ngram）互补，可叠加使用。
- **代价模型指导**：提供清晰的盈亏平衡条件，帮助实践者快速评估应用收益。
- **开源代码**：已提供 GitHub 仓库，便于复现和扩展。

## 8. 不足与局限

- **仅支持只读工具**：目前只对标记为只读的工具进行推测（如搜索、浏览），对于写操作（如数据库写入、文件修改）不支持，因为推测失败后的回滚机制仍待未来工作（沙箱或状态检查点）。
- **依赖 open-weight 模型和引擎特性**：需要后端提供 streaming completions、per-token logprobs 和 prefix cache 共享，闭源 API（如 OpenAI GPT-4）无法直接部署 D1 和 D2。
- **高负载下收益缩减**：实验主要面向延迟优化的轻负载场景（单用户或少量并发），在重度批处理下探针会抢占 foreground 解码资源，重叠收益降低。
- **模型格式边界**：若模型原生工具调用格式与强制探针前缀不匹配（如 Qwen3.5 原生 XML 格式），则 α 接近 0，自推测失效；需改用 JSON 格式提示才能工作。
- **小模型增益有限**：Qwen3-4B 虽然在 tau2 上有 1.15× 平均加速，但在 GAIA 和 HotpotQA 上基本持平（1.00–1.03×），因为小模型解码时 CoT 窗口短、解码快，重叠窗口过小。
- **延迟分解中的未覆盖**：论文未系统分析不同网络延迟下的实时变化（仅 tau2 可控模拟），且实验结果只在一类 GPU（H20）上获得，未在更常见架构（如 A100、H100）上验证。
- **控制器的额外开销**：尽管声称开销很小（≤0.05s Python 层），但在极低延迟任务中可能仍不可忽略。
