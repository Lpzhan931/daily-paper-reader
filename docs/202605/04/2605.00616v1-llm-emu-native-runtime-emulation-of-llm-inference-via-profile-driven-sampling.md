## 摘要
对大语言模型（LLM）服务系统的真实评估需要在线工作负载、动态到达、排队以及服务引擎用于执行批处理的本地调度，但在 GPU 上运行此类实验成本高昂。现有的模拟器虽然降低了成本，但通常以离线或时间扭曲模式运行，需要重新实现服务引擎调度器，或者依赖准确的算子/内核级延迟模型。我们提出了 LLM-Emu，这是一种针对 vLLM 的服务原生仿真器，它保留了生产环境中的 HTTP、调度、KV 缓存和输出处理路径，仅将 GPU 前向执行替换为基于剖析采样的延迟和合成输出 Token。在两种不同的 GPU、四种模型变体、两个模型系列、两个注意力后端以及泊松和突发 ShareGPT 工作负载上的测试表明，LLM-Emu 能够紧密追踪真实的 vLLM 服务行为：TPOT 和 ITL 的绝对误差保持在 4.8% 以内，端到端（E2E）延迟在 5.3% 以内，输出吞吐量在 1.9% 以内；TTFT 波动较大，最大误差为 10.4%，反映了其对准入和队列状态的敏感性。这些结果表明，轻量级的服务原生仿真可以支持 LLM 服务系统的实际在线实验。LLM-Emu 已在 https://github.com/AKafakA/llm-emu 开源。

## Abstract
Realistic evaluation of LLM serving systems requires online workloads, dynamic arrivals, queueing, and the serving engine's local scheduling for execution batching, but running such experiments on GPUs is expensive. Existing simulators reduce this cost, but often operate offline or in time-warped mode, re-implement serving-engine schedulers, or require accurate operator/kernel-level latency models. We present LLM-Emu, a serving-native emulator for vLLM that preserves the production HTTP, scheduling, KV-cache, and output-processing paths while replacing only GPU forward execution with profile-sampled latency and synthetic output tokens. Tested on two different GPUs, four model variants, two model families, two attention backends, and both Poisson and bursty ShareGPT workloads, LLM-Emu closely tracks real vLLM serving behavior: TPOT and ITL stay within $4.8\%$ absolute error, E2E latency within $5.3\%$, and output throughput within $1.9\%$; TTFT is less stable, with maximum error $10.4\%$, reflecting its sensitivity to admission and queue state. These results suggest that lightweight, serving-native emulation can support practical online experimentation for LLM-serving systems. LLM-Emu is open sourced at https://github.com/AKafakA/llm-emu.

### 2. 方法论
*   **核心思想**：**“逻辑保留，执行替换”**。LLM-Emu 作为一个插件嵌入到 vLLM 进程中，保留其 HTTP 栈、调度器、KV 缓存管理和输出处理路径，仅将 GPU 的前向计算（Forward Execution）替换为基于离线剖析（Profile）采样的延迟模拟。
*   **关键技术细节**：
    *   **密度感知延迟预测器（Oracle）**：使用一个二维剖析表，以“当前批次的 Token 总数”和“并发请求数”为索引。
    *   **邻域池化算法（Algorithm 1）**：针对剖析数据稀疏的区域，采用密度感知的最近邻扩展（Shepard 权重采样），确保在各种批次形状下都能获得可靠的延迟预测。
    *   **定时器解析的 Future 机制**：为了模拟异步调度中“调度器与工作进程”的重叠，LLM-Emu 使用基于定时器的异步 Future，在预测的延迟到达后返回合成的 Token ID。
    *   **离线剖析**：通过在真实 GPU 上运行不同速率的工作负载，收集每一步（Iteration）的延迟分布，生成 JSON 格式的剖析包。

### 6. 主要结论与发现
*   **高精度仿真**：在稳态指标上，LLM-Emu 与真实 GPU 执行高度一致。
    *   **TPOT（每 Token 延迟）和 ITL（迭代延迟）**：绝对误差 < 4.8%。

### 7. 优点
*   **原生集成**：直接作为 vLLM 插件运行，无需重写调度逻辑，能够自动跟随 vLLM 的版本更新（如 Chunked Prefill 等新特性）。
*   **低维护成本**：不需要复杂的算子级建模或 CUDA 拦截，只需更换 JSON 剖析文件即可支持新硬件或新模型。
*   **真实感强**：支持真实的 HTTP 客户端接入，可直接用于测试上层应用的性能。

### 8. 不足与局限
*   **单节点限制**：目前的验证仅限于单节点部署，尚未在多 GPU 张量并行（TP）或多节点分布式环境下进行充分验证。
*   **剖析成本**：虽然仿真不需要 GPU，但前期收集剖析数据仍需要数小时的 GPU 时间。
*   **工作负载耦合**：目前的采样逻辑在剖析数据与评估数据分布一致时表现最好，对于完全未见过的极端工作负载，预测精度可能下降。
*   **API 漂移风险**：虽然代码量小，但仍需针对 vLLM 的内部 API 变化进行少量维护。
