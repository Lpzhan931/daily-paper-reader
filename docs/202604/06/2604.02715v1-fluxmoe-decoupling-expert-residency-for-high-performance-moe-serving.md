## 摘要
混合专家（MoE）模型已成为扩展大语言模型的主流范式，但其快速增长的参数规模在推理过程中引入了根本性的低效：大多数专家权重在 GPU 显存中处于闲置状态，同时与键值（KV）缓存等对性能至关重要的运行时状态竞争资源。由于 KV 缓存容量直接决定了推理服务的吞吐量，这种不匹配导致了显存利用不足和性能下降。在本文中，我们提出了 FluxMoE，这是一种新型 MoE 推理系统，它将专家参数从持久的 GPU 驻留中解耦。FluxMoE 引入了一种专家分页（expert paging）抽象，将专家权重视为流式的瞬时资源，按需实例化并在使用后立即驱逐，从而允许 GPU 显存优先分配给对吞吐量至关重要的运行时状态。我们在 vLLM 之上实现了 FluxMoE，以在严苛的显存限制下实现高效的 MoE 推理。实验结果表明，在显存密集型场景下，FluxMoE 在不损害模型保真度的情况下，相比 vLLM 实现了高达 3.0 倍的吞吐量提升。

## Abstract
Mixture-of-Experts (MoE) models have become a dominant paradigm for scaling large language models, but their rapidly growing parameter sizes introduce a fundamental inefficiency during inference: most expert weights remain idle in GPU memory while competing with performance-critical runtime state such as the key-value (KV) cache. Since KV cache capacity directly determines serving throughput, this mismatch leads to underutilized memory and degraded performance. In this paper, we present FluxMoE, a new MoE inference system that decouples expert parameters from persistent GPU residency. FluxMoE introduces an expert paging abstraction that treats expert weights as streamed, transient resources, materializing them on demand and evicting them immediately after use, allowing GPU memory to be preferentially allocated to throughput-critical runtime state. We implement FluxMoE atop vLLM to enable efficient MoE inference under severe memory constraints. Experimental results demonstrate that FluxMoE achieves up to 3.0$\times$ throughput gains over vLLM in memory-intensive regimes, without compromising model fidelity.

## 1. 核心问题与整体含义
混合专家模型（MoE）通过增加专家数量实现了模型能力的飞跃，但其庞大的参数量（如 DeepSeek-R1 的 671B 参数）给推理带来了巨大的显存压力。
*   **研究动机**：在现有的推理框架（如 vLLM）中，模型权重被视为“持久驻留”在 GPU 显存中的资产。然而，MoE 在每一层执行时仅激活极少数专家，绝大多数权重在显存中处于闲置状态。
*   **核心矛盾**：闲置的专家权重与对吞吐量至关重要的 **KV Cache（键值缓存）** 激烈竞争显存空间。KV Cache 容量受限直接导致 Batch Size 无法扩大，从而限制了推理服务的吞吐量。
*   **整体含义**：FluxMoE 提出将专家参数从 GPU 持久驻留中解耦，将其视为“流式瞬态资源”，通过按需加载和即时驱逐，将显存空间优先让渡给 KV Cache。

## 2. 方法论
FluxMoE 的核心思想是**专家分页（Expert Paging）**，通过以下三个关键技术实现：

1.  **PagedTensor（张量分页）**：
    *   **虚拟化抽象**：利用 CUDA 虚拟内存管理（VMM）API，为所有专家张量预留稳定的虚拟地址空间，但物理上仅映射一个小的环形缓冲区（通常只需容纳当前层和预取层）。
    *   **透明性**：由于虚拟地址固定，现有的计算算子（PyTorch/Triton）无需修改即可访问动态加载的权重。
    *   **异步生命周期**：通过 CUDA Stream 和 Event 强制执行 WAR（写后读）和 RAW（读后写）约束，确保计算与 I/O 完全重叠。

2.  **带宽平衡存储层级（Bandwidth-Balanced Storage Hierarchy）**：
    *   **多级存储**：将专家权重分布在“压缩 GPU 显存”和“主机 DRAM”中。
    *   **无损压缩**：针对 MoE 权重指数位（Exponent）分布集中的特点，采用无损 Huffman 编码，节省约 20% 空间，并在 GPU 上实时解压。
    *   **比例放置策略**：根据各存储介质的有效带宽（如 PCIe 带宽 vs. GPU 解压带宽）按比例分配参数，确保各后端同步完成加载，消除 I/O 瓶颈。

3.  **预算感知驻留规划器（Budget-Aware Residency Planner）**：
    *   **闭环控制**：动态监控计算时间（$\tau_{comp}$）与加载时间（$\tau_{load}$）的比值 $\rho$。
    *   **动态调整**：若计算时间充裕（$\rho > 1$），则减少 GPU 驻留比例以释放显存给 KV Cache；若加载变慢（$\rho < 0.9$），则增加驻留比例。

*   **硬件依赖性**：系统的性能高度依赖 PCIe 带宽。在 PCIe 3.0 或带宽受限的环境下，流式传输可能成为无法隐藏的瓶颈。
