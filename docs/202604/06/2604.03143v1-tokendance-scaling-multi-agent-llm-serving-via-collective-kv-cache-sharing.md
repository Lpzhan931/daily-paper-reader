## 摘要
多智能体大语言模型（LLM）应用通常以同步轮次的方式组织执行，由中央调度器收集所有智能体的输出并重新分发合并后的上下文。这种全收集（All-Gather）通信模式产生了海量的 KV 缓存冗余，因为每个智能体的提示词都包含相同的共享输出块，而现有的重用方法无法高效地利用这一特性。我们提出了 TokenDance，该系统通过利用全收集模式进行集合式 KV 缓存共享，从而扩展了并发智能体的数量。TokenDance 的 KV 收集器（KV Collector）通过单个集合步骤在整个轮次内执行 KV 缓存重用，因此无论智能体数量多少，重用共享块的开销仅需支付一次。其差异感知存储（Diff-Aware Storage）将同级缓存编码为相对于单个主副本的块稀疏差异，在代表性工作负载上实现了 11-17 倍的压缩。在 GenerativeAgents 和 AgentSociety 上的评估表明，在满足服务水平目标（SLO）要求的前提下，TokenDance 支持的并发智能体数量比带有前缀缓存的 vLLM 高出 2.7 倍，将单个智能体的 KV 缓存存储降低了高达 17.5 倍，并比针对单个请求的位置无关缓存实现了高达 1.9 倍的预填充加速。

## Abstract
Multi-agent LLM applications organize execution in synchronized rounds where a central scheduler gathers outputs from all agents and redistributes the combined context. This All-Gather communication pattern creates massive KV Cache redundancy, because every agent's prompt contains the same shared output blocks, yet existing reuse methods fail to exploit it efficiently. We present TokenDance, a system that scales the number of concurrent agents by exploiting the All-Gather pattern for collective KV Cache sharing. TokenDance's KV Collector performs KV Cache reuse over the full round in one collective step, so the cost of reusing a shared block is paid once regardless of agent count. Its Diff-Aware Storage encodes sibling caches as block-sparse diffs against a single master copy, achieving 11-17x compression on representative workloads. Evaluation on GenerativeAgents and AgentSociety shows that TokenDance supports up to 2.7x more concurrent agents than vLLM with prefix caching under SLO requirement, reduces per-agent KV Cache storage by up to 17.5x, and achieves up to 1.9x prefill speedup over per-request position-independent caching.

以下是对该论文的结构化总结：

### 1. 核心问题与研究动机
*   **核心问题**：在多智能体应用（如社会模拟、协作编码）中，智能体通常按“轮次”执行。每一轮结束时，调度器会收集所有智能体的输出并分发给所有智能体作为下一轮的上下文（即 **All-Gather 模式**）。
*   **研究动机**：
    *   **结构性冗余**：虽然所有智能体在下一轮接收到的共享信息基本一致，但由于每个智能体拥有不同的“私有历史”，导致共享块在 Prompt 中的绝对位置不同。
    *   **现有技术局限**：传统的前缀缓存（Prefix Caching）因位置偏移失效；位置无关缓存（PIC）虽然能重用，但对 $N$ 个智能体需要重复执行 $N$ 次重用计算，且每个智能体仍需存储一份完整的 KV 缓存副本，导致显存迅速耗尽。

### 2. 方法论
TokenDance 的核心思想是将优化粒度从“单个请求”提升到“多智能体轮次”，通过以下关键技术实现：
*   **轮次感知提示词接口（Round-Aware Prompt Interface）**：在逻辑块之间插入特殊的隔离符（`<TTSEP>`），使运行时系统能识别块边界，并采用基于段（Segment）的哈希而非固定长度哈希，从而在不同偏移量下识别共享内容。
*   **集合式 KV 缓存重用（Collective KV Cache Reuse）**：
    *   **合并计算**：将同一轮次内的 $N$ 个请求分组，对共享块仅执行一次 RoPE 旋转和“重要位置”选择分析，将计算开销从 $O(N)$ 降至接近 $O(1)$。
    *   **层级同步**：在模型执行的每一层中，通过批处理方式同时处理组内所有请求的重用逻辑。
*   **差异感知存储（Diff-Aware Storage）**：
    *   **主从布局（Master-Mirror Layout）**：仅存储一个完整的“主副本（Master）”，其余智能体的缓存仅存储相对于主副本的“块稀疏差异（Sparse Diffs）”。
    *   **融合恢复（Fused Diff Restore）**：在将缓存从内存/显存传输到计算单元的流水线中，直接应用差异补丁，避免了额外的显存读写开销。

### 6. 主要结论与发现
*   **容量提升**：在相同的延迟 SLO 下，TokenDance 支持的并发智能体数量比 vLLM 提升了 **2.7 倍**。
*   **存储优化**：相比于为每个智能体存储独立缓存，TokenDance 将单个智能体的存储开销降低了 **94%**（在 14B 模型上实现 17.5 倍压缩）。
*   **性能加速**：预填充（Prefill）阶段相比于传统的 PIC 方法实现了高达 **1.9 倍** 的加速，端到端延迟降低了 **2.3 倍**。
*   **准确性**：TokenDance 不会引入超出底层 PIC 方法之外的额外精度损失。

### 7. 优点
*   **应用层感知**：敏锐捕捉到了多智能体应用中特有的 All-Gather 通信模式，并将其转化为底层系统的优化机会。
*   **计算与存储协同**：不仅通过集合计算减少了重复的 RoPE 旋转，还通过主从布局解决了显存瓶颈，双管齐下。
*   **工程落地性强**：通过融合内核（Fused Kernel）减少了数据搬运，确保了压缩带来的收益不会被解压开销抵消。

### 8. 不足与局限
*   **模式依赖**：该系统高度依赖于 All-Gather 这种同步轮次模式。如果多智能体应用采用完全异步或点对点的通信模式，其收益将大幅下降。
*   **精度风险**：由于底层使用了选择性重计算（基于 CacheBlend），在极长链条的推理中，微小的数值差异可能会随轮次累积，导致智能体行为发生漂移（虽然文中验证了短期内的稳定性）。
*   **硬件限制**：目前主要在单卡上验证，尚未详细讨论在多卡分布式环境下的集合通信开销与缓存共享的权衡。
