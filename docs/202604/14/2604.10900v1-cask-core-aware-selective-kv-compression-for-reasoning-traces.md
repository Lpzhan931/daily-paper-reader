## 摘要
在大语言模型执行长文本推理时，KV 缓存随解码长度迅速增长，导致内存和推理稳定性出现瓶颈。现有的面向推理的 KV 压缩大多遵循以逐出为中心的视角：即更准确地估计 token 的重要性，然后丢弃排名较低的条目。我们的分析表明，仅靠优化评分器往往无法实质性地重组实际保留集，因此可能不是保持推理行为的主要手段。相反，我们将推理 KV 压缩定义为一个保持行为的结构化整合问题。CASK 将解码阶段的推理轨迹划分为受保护的核心（core）和具有高冗余度的可合并草稿（scratch），前者用于锚定答案生成和中间状态。核心部分被保留，而选择性整合仅应用于草稿部分。为了应对提示词密集（prompt-heavy）的情况（即前缀可能在解码阶段压缩生效前就耗尽预算），CASK 进一步采用了两阶段设计：前缀逐出后接解码阶段整合。在 H100 推理基准测试中，CASK 在 AIME24 和 AIME25 上相同预算下的全量 KV 续写保真度均高于 TriAttention，且多次出现 cask@384 > triattention@512 的性能超越。在提示词密集的重放实验中，multi_news 和 vcsum 验证了解码阶段的活性，而 qmsum 和 gov_report 则暴露了前缀预算耗尽的边界。总体证据支持一个简单的结论：有效的推理 KV 压缩与其说依赖于更复杂的评分器工程，不如说依赖于将核心保留与选择性草稿整合相结合，从而降低可用预算的边界。

## Abstract
In large language models performing long-form reasoning, the KV cache grows rapidly with decode length, creating bottlenecks in memory and inference stability. Existing reasoning-oriented KV compression has mostly followed an eviction-centered view: estimate token importance more accurately, then discard lower-ranked entries. Our analysis suggests that scorer refinement alone often fails to substantially reorganize the actual keep-set and may therefore not be the main lever for preserving reasoning behavior. We instead frame reasoning KV compression as a behavior-preserving structured consolidation problem. CASK partitions the decode-time reasoning trace into a protected core that anchors answer formation and intermediate state, and mergeable scratch with high redundancy. The core is preserved, while selective consolidation is applied only to the scratch. To address prompt-heavy regimes where the prefix can exhaust the budget before decode-stage compression becomes active, CASK further uses a two-stage design: prefix eviction followed by decode-stage consolidation. On the H100 reasoning gate, CASK shows higher full-KV continuation fidelity than TriAttention at matched budgets on both AIME24 and AIME25, with recurring cask@384 > triattention@512 crossings. In prompt-heavy replay, multi_news and vcsum act as decode-active witnesses, while qmsum and gov_report expose the prefix_budget_exhausted boundary. The overall evidence supports a simple conclusion: effective reasoning KV compression depends less on more elaborate scorer engineering than on combining core preservation with selective scratch consolidation to lower the usable budget frontier.

### 论文总结：CASK - 面向推理轨迹的核心感知选择性 KV 压缩

#### 1. 核心问题与研究动机
*   **核心问题**：在大型语言模型（LLM）执行长链推理时，KV 缓存（Key-Value Cache）随解码长度呈线性增长，导致内存瓶颈和推理稳定性下降。
*   **研究动机**：
    *   **现有方法局限**：目前的推理压缩方法（如 TriAttention, R-KV）多采用“逐出（Eviction）”策略，即通过评分器对 Token 重要性排序并丢弃低分 Token。作者发现，单纯优化评分器很难实质性改变保留集（Keep-set），无法从根本上解决推理行为的保持问题。
    *   **推理轨迹特性**：推理过程中存在“角色差异”。有些 Token 是锚定答案的核心（Core），而有些是冗余的探索性草稿（Scratch）。简单地丢弃非核心 Token 会导致状态坍塌。
    *   **目标**：将压缩问题从“排名丢弃”转变为“结构化整合”，在降低内存占用的同时，最大限度地保留模型的推理行为。

#### 2. 方法论（CASK）
*   **核心思想**：将解码期的推理轨迹划分为**受保护的核心（Protected Core）**和**可合并的草稿（Mergeable Scratch）**。核心部分绝对保留，草稿部分进行选择性合并（Consolidation）。
*   **关键技术细节**：
    *   **核心/草稿分解**：核心集包含直接贡献于答案形成、中间状态锚定和近期推理转折的 Token；草稿集包含重复探索、自我检查和近乎重复的推导。
    *   **两阶段压缩（Two-stage Compression）**：
        *   **第一阶段（前缀逐出）**：针对 Prompt 较重的情况，先对前缀进行逐出，为解码阶段腾出预算空间。
        *   **第二阶段（解码整合）**：在解码过程中应用核心保护和草稿的选择性合并。
    *   **代表集构建（Representative-set）**：在满足预算 $B$ 的前提下，确保核心集 $C$ 包含在最终代表集 $R$ 中，通过合并草稿 Token 来构造代表性状态。
    *   **Kappa 加权合并几何**：利用频率感知几何（基于 Phase 1 的研究成果）来衡量 Token 间的距离，优先合并对未来查询影响较小的频段差异。
    *   **m-折叠构造（m-folding）**：合并并非简单丢弃，而是通过加权平均（保留组质量）来构造代表性的 Key 和 Value。

*   **Benchmark（对比方法）**：主要对比 **TriAttention**（当时最先进的面向推理的 KV 压缩方法）。

#### 6. 主要结论与发现
*   **性能超越**：在相同预算下，CASK 的保真度系统性地高于 TriAttention。特别是在 AIME 任务中，`cask@384` 的表现多次超过了 `triattention@512`。
*   **预算边界下移**：CASK 的核心价值不在于更激进地丢弃 Token，而是在更低的预算下维持与全量 KV 相同的行为（即降低了“可用预算前沿”）。
*   **角色分离的有效性**：实验证明，仅靠核心保护是不够的，必须结合草稿部分的代表性重构才能维持高保真度。
*   **两阶段设计的必要性**：在提示词密集的场景下，两阶段设计防止了前缀占用全部预算而导致解码压缩失效的问题。

#### 7. 优点
*   **视角创新**：从单纯的“重要性排序”转向“角色感知策略”，更符合推理任务的本质。
*   **评估严谨**：引入教师强制重放协议，能够精细观察行为漂移的发生时机，比单纯看最终答案准确率更具说服力。
*   **两阶段架构**：解决了长上下文任务中前缀与解码阶段压缩冲突的实际工程问题。
*   **透明度高**：论文详细记录了失败的尝试（Phase 1）及其转向的逻辑，增加了研究的可信度。

#### 8. 不足与局限
*   **覆盖范围**：虽然在多个任务上表现优异，但尚未覆盖全部主流基准测试，且在某些多跳问答（如 2wikimqa）中优势不明显。
