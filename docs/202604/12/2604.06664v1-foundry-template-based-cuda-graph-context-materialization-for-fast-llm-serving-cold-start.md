## 摘要
现代大语言模型（LLM）服务提供商日益依赖自动扩缩容和并行重配置来应对快速变化的工作负载，但冷启动延迟仍是一个主要瓶颈。虽然最近的系统已将模型权重加载时间缩短至秒级，但 CUDA 图捕获（capture）仍需数十秒到数分钟，且往往在启动耗时中占据主导地位。不幸的是，CUDA 图无法进行简单的序列化：除了图拓扑结构外，它们还与执行上下文紧密耦合，包括嵌入在算子（kernel）参数中的设备地址以及在预热期间延迟加载的算子代码。现有方法要么依赖于脆弱的针对特定算子的补丁，要么依赖于重量级的进程级检查点/恢复（checkpoint/restore），在应对动态并行切换时缺乏灵活性。我们提出了 Foundry，这是一个基于模板的 CUDA 图上下文实例化系统，它在离线处理阶段持久化图拓扑和执行上下文，并以极低的开销在线重建可执行图。Foundry 强制执行确定性内存布局，自动提取并重新加载捕获图所需的算子二进制文件，并通过基于拓扑的模板化降低在线重建成本。对于分布式服务，Foundry 进一步实现了单 GPU 离线捕获，通过仅修补与 rank 相关的通信状态，即可为多 GPU 部署生成模板。在参数量高达 235B 的稠密和 MoE 模型上，Foundry 将冷启动延迟降低了高达 99%，将 Qwen3-235B-A22B 的初始化时间从 10 分钟缩短至 3.9 秒，同时保留了 CUDA 图带来的吞吐量提升。

## Abstract
Modern LLM service providers increasingly rely on autoscaling and parallelism reconfiguration to respond to rapidly changing workloads, but cold-start latency remains a major bottleneck. While recent systems have reduced model weight loading to seconds, CUDA graph capture still takes tens of seconds to minutes and often dominates startup. Unfortunately, CUDA graphs cannot be naively serialized: beyond graph topology, they are tightly coupled to execution context, including device addresses embedded in kernel arguments and kernel code lazily loaded during warmup. Existing approaches either rely on brittle kernel-specific patching or heavyweight process-level checkpoint/restore that are inflexible to dynamic parallelism switching. We present Foundry, a template-based CUDA graph context materialization system that persists both graph topology and execution context during an offline processing stage, and reconstructs executable graphs online with negligible overhead. Foundry enforces deterministic memory layouts, automatically extracts and reloads kernel binaries required by captured graphs, and reduces online reconstruction costs through topology-based templating. For distributed serving, Foundry further enables a single-GPU offline capture to generate templates for multi-GPU deployments by patching only rank-dependent communication state. Across dense and MoE models up to 235B parameters, Foundry reduces cold-start latency by up to 99%, cutting the initialization time of Qwen3-235B-A22B from 10 minutes to 3.9 seconds while preserving the throughput gains of CUDA graphs.

### Foundry：基于模板的 CUDA 图上下文实例化，助力大语言模型服务快速冷启动

#### 1. 核心问题与整体含义（研究动机和背景）
在现代大语言模型（LLM）推理服务中，为了应对波动的流量，服务商依赖自动扩缩容（Autoscaling）和并行策略动态切换。然而，**冷启动延迟**是提升资源利用率的主要障碍。
*   **背景**：虽然模型权重的加载已通过 RDMA 等技术缩短至秒级，但 **CUDA 图（CUDA Graph）的捕获（Capture）过程**依然极其耗时（从数十秒到数分钟不等），成为冷启动的性能瓶颈。
*   **核心挑战**：CUDA 图无法直接序列化存储，因为它与执行上下文深度绑定。图中嵌入了特定的 GPU 内存地址（指针）和内核函数句柄（Kernel Handles），这些在每次运行或不同设备上都会发生变化。
*   **论文目标**：提出 Foundry 系统，通过“上下文实例化”技术，实现 CUDA 图的离线持久化与在线极速重建。

#### 2. 方法论：核心思想与关键技术
Foundry 的核心思想是**解耦图拓扑与执行上下文**，并在加载时重新建立一致的运行环境。
*   **确定性内存布局（Deterministic Memory Layout）**：
    *   拦截 CUDA 驱动的虚拟内存管理（VMM）API。
    *   将所有内存分配重定向到预留的连续虚拟地址空间，确保在线加载时的内存地址与离线捕获时完全一致，从而避免了复杂的指针修补（Patching）。
*   **内核二进制提取与重载（Binary Extraction & Reload）**：
    *   拦截内核模块加载 API，自动提取并哈希存储内核二进制文件（cubin）。
    *   在线阶段直接重载这些二进制文件并恢复函数句柄，无需重新执行预热（Warmup）或 JIT 编译。
*   **基于拓扑的模板化（Topology-Based Templating）**：
    *   **组内共享**：发现不同 Batch Size 的图往往具有相同的拓扑结构。Foundry 为每种拓扑创建一个“模板”，在线时仅通过 `cuGraphExecUpdate` 更新节点参数（如 Grid 维度、参数缓冲区），避免了昂贵的图实例化开销。
    *   **跨 Rank 共享**：在分布式推理（DP/EP）中，通过 Stub 层模拟通信，实现单 GPU 离线捕获即可生成适用于所有 Rank 的模板，显著降低离线处理成本。

#### 6. 主要结论与发现
*   **极速启动**：Foundry 将冷启动延迟降低了 **95%–99%**。
*   **超大规模模型优势**：对于 Qwen3-235B-A22B，初始化时间从 **10 分钟缩短至 3.9 秒**。

*   **存储高效**：Foundry 的归档文件比进程级检查点小 4–5 倍（例如 14B 模型仅需 1.1GB）。

#### 7. 优点
*   **内核无关性**：无需像 Medusa 等前作那样为每个算子编写手动修补规则，具有极强的通用性。
*   **分布式友好**：支持单卡捕获、多卡部署，解决了分布式环境下 CUDA 图重构的难题。
*   **异步优化**：利用确定性内存预分配，使图构建可以与模型权重加载等任务并行，进一步压榨启动时间。

#### 8. 不足与局限
*   **流水线并行（PP）限制**：目前不支持流水线并行，因为不同阶段（Stage）的图拓扑不一致，无法简单共享。
*   **离线依赖**：仍需要一个一次性的离线处理阶段来生成归档文件，对于频繁变动的模型结构可能需要重新处理。
