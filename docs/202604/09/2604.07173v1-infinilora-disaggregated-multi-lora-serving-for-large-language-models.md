## 摘要
LoRA 实现了大语言模型（LLM）的高效定制，并广泛应用于多租户和多任务服务中。然而，诸如混合专家模型（MoE）等新兴模型架构显著增加了 LoRA 的内存开销，导致现有的耦合式 LoRA 服务设计扩展性差，且容易引发尾部延迟膨胀。我们提出了 InfiniLoRA，这是一种解耦式 LoRA 服务系统，将 LoRA 执行与基础模型推理相分离。InfiniLoRA 引入了一个共享的 LoRA 服务器，具备并行感知执行、服务水平目标（SLO）驱动的资源配置以及关键路径优化，包括 GPU 发起的通信和硬件专门化的 LoRA 内核。实验表明，在严格的延迟 SLO 约束下，InfiniLoRA 可使可服务请求率平均提高 3.05 倍，并将满足 SLO 要求的 LoRA 适配器比例提高 54.0%。

## Abstract
LoRA enables efficient customization of LLMs and is widely used in multi-tenant and multi-task serving. However, emerging model architectures such as MoE significantly increase LoRA memory cost, making existing coupled LoRA serving designs poorly scalable and prone to tail-latency inflation. We present InfiniLoRA, a disaggregated LoRA serving system that decouples LoRA execution from base-model inference. InfiniLoRA introduces a shared LoRA Server with parallelism-aware execution, SLO-driven provisioning, and critical-path optimizations, including GPU-initiated communication and hardware-specialized LoRA kernels. Experiments show that InfiniLoRA can achieve an average $3.05\times$ increase in serviceable request rate under strict latency SLOs, and improve the percentage of LoRA adapters satisfying the SLO requirement by 54.0\%.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）向混合专家模型（MoE）架构演进，LoRA 适配器的参数量显著增加（因为每个专家都需要对应的 LoRA 权重）。在现有的“耦合式”服务架构中，LoRA 适配器与基础模型权重、KV 缓存共同挤占 GPU 显存。这导致了两个核心矛盾：
*   **显存瓶颈：** MoE 模型的 LoRA 适配器极大，单卡能缓存的适配器数量锐减，导致频繁的换入换出，严重推高了首字延迟（TTFT）的尾部延迟。
*   **扩展性差：** 简单的“水平扩展”（增加实例）会重复存储基础模型权重，浪费显存；“垂直扩展”（单实例多卡）则会增加通信开销并降低硬件利用率。
**InfiniLoRA 的核心动机**是实现 LoRA 执行与基础模型推理的**解耦（Disaggregation）**，通过独立的 LoRA 服务器集群来管理和计算 LoRA，从而打破显存限制。

### 2. 方法论：核心思想与关键技术
InfiniLoRA 提出了一种解耦式架构，将 LoRA 适配器从 LLM 推理实例中分离出来，由专门的 **LoRA Server** 统一管理。其关键技术包括：
*   **并行感知执行（Parallelism-Aware Execution）：** 提出了混合并行策略（$EP_x-PP_y$），结合了专家并行（EP）和流水线并行（PP）。通过在 LoRA Server 内部对适配器进行分片，平衡了通信粒度、同步开销和计算效率。
*   **SLO 驱动的资源配置：** 建立了一个概率模型（基于泊松分布和动态规划），将 TTFT 的 SLO 要求转化为“即时准入率（IAR）”。通过二分查找和算法计算出满足特定 SLO（如 95% 的请求不排队）所需的最小 LoRA 缓存容量和 GPU 数量。
*   **关键路径优化：**
    *   **GPU 发起的通信：** 利用 IBGDA 技术实现单边 RDMA 推送（Push-based），绕过 CPU 宿主机，降低通信延迟。
    *   **硬件专用内核：** 针对 Hopper 架构优化了 BGMV 和 SGMV 内核，利用 TMA 和 WGMMA 指令提升计算速度。
    *   **层级异步加载：** 在推理过程中以“层”为粒度流水化加载 LoRA 权重，并结合调度器驱动的预取技术消除冷启动停顿。

### 6. 主要结论与发现
*   **吞吐量提升：** 在严格的延迟 SLO 约束下，InfiniLoRA 的平均可服务请求率比 S-LoRA 提升了 **3.05 倍**。

*   **解耦的必要性：** 实验证明，单纯的解耦（不加优化）反而会因为通信开销降低性能，只有结合关键路径优化（RDMA、专用内核等），解耦的优势才能体现。
*   **缓存是瓶颈：** 在解耦架构中，显存容量（Cache Capacity）是影响尾部延迟的主要因素，而计算能力通常有富余。

### 7. 优点
*   **架构创新：** 首次针对 MoE 时代的 LoRA 服务提出了彻底的计算与存储解耦方案。
*   **数学严谨性：** 提供了基于概率模型的资源配置算法，使系统能够根据 SLO 目标自动调整资源。
*   **底层优化深入：** 深入到 GPU 硬件指令集和 RDMA 通信协议层面，解决了解耦带来的额外延迟问题。

### 8. 不足与局限
*   **网络依赖性：** 该系统高度依赖高性能互联（如 InfiniBand 或 NVLink），在低带宽网络环境下，解耦带来的通信开销可能会抵消其显存收益。
*   **模型适用性：** 主要针对 LoRA 权重巨大的 MoE 模型。对于适配器极小的稠密模型（Dense Models），解耦带来的复杂度和通信开销可能并不划算。
*   **系统复杂度：** 引入了独立的服务器节点，增加了分布式系统的运维和故障处理难度。
