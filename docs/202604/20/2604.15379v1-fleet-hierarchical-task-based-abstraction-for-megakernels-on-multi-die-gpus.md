## 摘要
现代 GPU 采用基于芯粒（chiplet）的设计，具有多个私有缓存层级，但目前的编程模型（CUDA/HIP）呈现的是扁平的执行层级，无法表达芯粒级的局部性或同步。这种不匹配导致了冗余的内存流量，并在大语言模型（LLM）推理等访存密集型工作负载中导致缓存利用率低下。我们提出了 Fleet，这是一种将计算映射到内存作用域的多级任务模型。Fleet 引入了芯粒任务（Chiplet-tasks），这是一种将工作和数据绑定到芯粒的新抽象，并能通过其共享的 L2 缓存实现协作。Wavefront 级、CU 级和设备级任务与现有抽象保持一致，而芯粒任务则揭示了层级结构中此前未被处理的一层。Fleet 被实现为一个具有每芯粒调度功能的持久化内核运行时，允许芯粒内的 worker 通过协调的缓存复用来协作执行任务。在搭载 Qwen3-8B 的 AMD Instinct MI350 上，通过持久化内核执行和每芯粒调度，Fleet 在 Batch Size 为 1-8 时实现的解码延迟比 vLLM 低 1.3-1.5 倍。在较大的 Batch Size 下，协作式权重分块（weight tiling）提高了 L2 命中率（Batch Size 为 32 时从 12% 提升至 54%，Batch Size 为 64 时从 39% 提升至 61%），将 HBM 流量减少了高达 37%，并相比于不感知芯粒的巨内核基准实现了 1.27-1.30 倍的加速。

## Abstract
Modern GPUs adopt chiplet-based designs with multiple private cache hierarchies, but current programming models (CUDA/HIP) expose a flat execution hierarchy that cannot express chiplet-level locality or synchronization. This mismatch leads to redundant memory traffic and poor cache utilization in memory-bound workloads such as LLM inference.   We present Fleet, a multi-level task model that maps computation to memory scopes. Fleet introduces Chiplet-tasks, a new abstraction that binds work and data to a chiplet and enables coordination through its shared L2 cache. Wavefront-level, CU-level, and device-level tasks align with existing abstractions, while Chiplet-tasks expose a previously unaddressed level of the hierarchy. Fleet is implemented as a persistent kernel runtime with per-chiplet scheduling, allowing workers within a chiplet to cooperatively execute tasks with coordinated cache reuse. On AMD Instinct MI350 with Qwen3-8B, Fleet achieves 1.3-1.5x lower decode latency than vLLM at batch sizes 1-8 through persistent kernel execution and per-chiplet scheduling. At larger batch sizes, cooperative weight tiling increases L2 hit rate (from 12% to 54% at batch size 32 and from 39% to 61% at batch size 64), reducing HBM traffic by up to 37% and delivering 1.27-1.30x speedup over a chiplet-unaware megakernel baseline.

## 论文详细总结（自动生成）

这篇论文由 AMD Research 团队提出，旨在解决现代多芯粒（Chiplet）GPU 架构与传统编程模型之间的失配问题。以下是对该论文的结构化总结：

### 1. 核心问题与研究动机
*   **硬件趋势与挑战**：现代 GPU（如 AMD MI300/350、NVIDIA Blackwell）已转向多芯粒设计，每个芯粒（AMD 称为 XCD）拥有私有的 L2 缓存。
*   **编程模型失配**：现有的 CUDA/HIP 编程模型（线程→Wavefront→工作组→网格）是扁平化的，无法表达“芯粒级”的数据局部性或同步需求。
*   **性能瓶颈**：在 LLM 推理的解码（Decode）阶段，由于无法控制工作组分配到特定芯粒，导致不同芯粒重复从 HBM 读取相同的权重数据，造成 L2 缓存利用率低下和冗余的内存流量。此外，频繁的内核启动（Kernel Launch）也带来了显著的延迟。

### 2. 方法论：Fleet 多级任务模型
Fleet 的核心思想是引入一个与硬件内存层级相匹配的**分层任务抽象**，并将其实现为**持久化巨内核（Persistent Megakernel）**。

*   **四级任务层级**：
    *   **Wavefront-task**：对应寄存器/LDS，处理逐元素操作（如 SiLU、残差相加）。
    *   **CU-task**：对应单个计算单元，处理注意力头、RMSNorm 等。
    *   **Chiplet-task（核心创新）**：跨越单个芯粒内的所有 worker，通过共享的 L2 缓存进行协作。它允许程序员指定数据分区和 L2 工作集。
    *   **Device-task**：协调 8 个芯粒共同完成一个算子（如完整的 GEMM）。
*   **关键技术细节**：
    *   **持久化内核运行时**：每个芯粒指定一个工作组作为“调度器（Scheduler）”，其余为“执行者（Worker）”。调度器在设备端直接分发任务，消除主机端启动开销。
    *   **协作式分块（Cooperative Tiling）**：采用 **M-major 遍历顺序**。确保同一芯粒内的多个 Worker 在短时间内访问相同的权重列，从而使第一个 Worker 加载的数据能被后续 Worker 在 L2 缓存中命中。
    *   **分层同步协议**：利用 L2 本地计数器进行芯粒内同步，仅在必要时（如芯粒间通信）才触发昂贵的跨芯粒一致性操作（如 `buffer_wbl2` 栅栏）。

### 3. 实验设计
*   **模型与场景**：使用 **Qwen3-8B** 稠密模型（bf16 精度）进行 LLM 解码推理测试。

*   **对比基准（Benchmark）**：
    1.  **vLLM (v0.17.2)**：代表目前最先进的生产级推理框架（采用标准算子内核启动）。

### 4. 资源与算力
*   **硬件**：明确使用了 **AMD Instinct MI350X** GPU。
*   **算力细节**：论文侧重于推理性能优化，未涉及模型训练时长。实验在单 GPU 上完成，利用了 MI350 的 5.3 TB/s HBM3 带宽和 256MB Infinity Cache。

### 5. 实验数量与充分性
*   **实验覆盖**：涵盖了从 Batch Size 1 到 64 的端到端延迟测试、L2 命中率分析、HBM 流量统计以及 Roofline 性能分析。
*   **充分性**：实验设计较为充分，通过消融实验清晰地拆解了“调度开销减少”与“L2 缓存复用”各自贡献的加速比。使用了硬件性能计数器（rocprofiler-sdk）进行底层数据采集，结果具有较高的客观性和说服力。

### 6. 主要结论与发现
*   **延迟降低**：在小 Batch Size (1-8) 下，Fleet 比 vLLM 快 **1.3–1.5 倍**，主要归功于持久化内核消除了内核启动开销。
*   **缓存优化**：在大 Batch Size (32-64) 下，协作式分块显著提升了 L2 命中率。例如在 BS=64 时，L2 命中率从 39% 提升至 **61.4%**。
*   **带宽节省**：通过 L2 复用，HBM 读取流量减少了高达 **37%**。
*   **加速比**：相比于不感知芯粒的巨内核基准，Fleet 在大 Batch Size 下实现了 **1.27–1.30 倍** 的额外加速。

### 7. 优点
*   **架构感知**：首次在持久化内核中系统性地引入了对多芯粒 GPU 内存层级的显式控制。
*   **显著减负**：通过软件管理的调度器，有效缓解了多芯粒架构带来的“类 NUMA”内存访问瓶颈。
*   **通用性**：提出的 Chiplet-task 抽象具有通用性，理论上可扩展至 NVIDIA Blackwell 等其他多芯粒架构。

### 8. 不足与局限
*   **实验范围限制**：目前仅在单 GPU 上对稠密模型（Qwen3-8B）进行了评估，未涉及多卡张量并行（TP）或混合专家模型（MoE）的复杂场景。
*   **寄存器压力**：巨内核将所有算子编译为一个函数，会导致寄存器压力增大，限制了每个 SIMD 上的活跃 Wavefront 数量，可能影响延迟隐藏能力。
*   **算子覆盖**：目前的 GEMM 实现尚未包含针对计算密集型场景（大 BS）的所有优化（如 K-splitting），导致在极高 Batch Size 下性能略逊于高度优化的库函数。
