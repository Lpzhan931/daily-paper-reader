## 摘要
大语言模型（LLM）推理正迅速成为核心数据中心服务，然而当前的推理服务栈仍将主机 CPU 置于编排和 Token 级控制的关键路径上。这使得 LLM 的性能对 CPU 干扰十分敏感，损害了应用的混部能力，并迫使运营商预留 CPU 余量，导致大量计算能力无法得到利用。我们提出了 Blink，这是一种端到端的推理服务架构，通过在 SmartNIC 和 GPU 之间重新分配职责，将主机 CPU 从稳态推理路径中移除。Blink 将请求处理卸载到 SmartNIC，由其通过 RDMA 将输入直接送入 GPU 内存，并使用持久化 GPU 内核取代主机驱动的调度，在无需 CPU 参与的情况下完成批处理、调度和 KV 缓存管理。实验评估表明，与 TensorRT-LLM、vLLM 和 SGLang 相比，即使在隔离环境下，Blink 的表现也优于所有基准系统，其预饱和 P99 TTFT 降低了高达 8.47 倍，P99 TPOT 降低了高达 3.40 倍，解码吞吐量提升了高达 2.1 倍，且每个 Token 的能耗降低了高达 48.6%。在存在 CPU 干扰的情况下，Blink 仍能保持稳定的性能，而现有系统的性能下降则高达两个数量级。

## Abstract
Large Language Model (LLM) inference is rapidly becoming a core datacenter service, yet current serving stacks keep the host CPU on the critical path for orchestration and token-level control. This makes LLM performance sensitive to CPU interference, undermining application colocation and forcing operators to reserve CPU headroom, leaving substantial capacity unutilized.   We introduce Blink, an end-to-end serving architecture that removes the host CPU from the steady-state inference path by redistributing responsibilities across a SmartNIC and a GPU. Blink offloads request handling to the SmartNIC, which delivers inputs directly into GPU memory via RDMA, and replaces host-driven scheduling with a persistent GPU kernel that performs batching, scheduling, and KV-cache management without CPU involvement.   Evaluated against TensorRT-LLM, vLLM, and SGLang, Blink outperforms all baselines even in isolation, reducing pre-saturation P99 TTFT by up to 8.47$\times$ and P99 TPOT by up to 3.40$\times$, improving decode throughput by up to 2.1$\times$, and reducing energy per token by up to 48.6$\%$. Under CPU interference, Blink maintains stable performance, while existing systems degrade by up to two orders of magnitude.

### 论文总结：Blink —— 脱离 CPU 的大语言模型推理架构

#### 1. 核心问题与整体含义（研究动机和背景）
当前主流的 LLM 推理服务栈（如 vLLM、TensorRT-LLM）在架构上存在一个根本性缺陷：**主机 CPU 仍处于编排和 Token 级控制的关键路径上**。
*   **研究动机**：自回归解码需要每生成一个 Token 就返回 CPU 进行调度、批处理管理和内核启动。这导致推理性能对 CPU 干扰（如数据中心内的“嘈杂邻居”任务）极其敏感。
*   **核心问题**：CPU 的干预不仅增加了延迟抖动，还迫使运营商预留大量 CPU 余量，导致资源利用率低下。论文旨在通过将服务栈委派给 SmartNIC 和 GPU，实现“无 CPU（CPU-Free）”的稳态推理。

#### 2. 方法论：核心思想与关键技术
Blink 的核心思想是将主机 CPU 转变为仅负责初始化的“配置平面”，而将稳态推理的“数据平面”完全移交给 SmartNIC（DPU）和 GPU。
*   **前端（DPU 侧）**：利用 BlueField-3 DPU 的 ARM 核心处理请求接入、Token 分词（Tokenizer）和请求调度。通过 **一侧 RDMA（One-sided RDMA）** 将输入直接写入 GPU 显存中的环形缓冲区，实现零拷贝。
*   **后端（GPU 侧）**：
    *   **持久化调度内核（Persistent Scheduler Kernel）**：在 GPU 上运行一个永不退出的 CUDA 内核，负责轮询环形缓冲区、管理连续批处理（Continuous Batching）和 Paged KV-cache。
    *   **设备端图形启动（Device-side Graph Launch）**：利用 CUDA Graph 技术，由持久化内核直接在 GPU 内部启动推理计算，完全绕过主机端的内核分发开销。
    *   **窗口化尾部启动恢复机制**：针对 CUDA 120 次连续启动的硬件限制，设计了窗口机制，通过原子操作重置启动计数，确保无限期的 Token 生成。

#### 3. 实验设计
*   **数据集/场景**：使用 **ShareGPT v3** 会话追踪数据集，模拟真实自然语言交互。
*   **测试模型**：涵盖不同规模和架构，包括 Llama-3 8B（稠密）、Phi-4 15B（稠密）、Qwen-3 32B（稠密）以及 Qwen-3 30B-A3B（MoE 混合专家模型）。
*   **Benchmark 与对比方法**：
    *   **基准系统**：TensorRT-LLM（使用相同推理引擎）、vLLM、SGLang。
    *   **测试维度**：隔离环境（Isolated） vs. 干扰环境（Colocated，使用 pbzip2 和 Ninja 编译任务模拟 CPU 高负载）。

*   **GPU**：单张 **NVIDIA H100 (96GB HBM3)**。
*   **CPU**：双路 Intel Xeon Gold 6336Y（共 96 核心）。
*   **SmartNIC/DPU**：NVIDIA **BlueField-3**（16 个 ARM Cortex-A78 核心，32GB 内存）。
*   **网络**：200 Gbps RDMA 连接。
*   **训练/运行**：论文侧重于推理服务，未提及模型训练时长，重点在于推理时的实时响应性能。

#### 5. 实验数量与充分性
*   **实验规模**：在 4 种模型上进行了 13 种不同请求负载（1-32 req/s）的测试，每组实验运行 60 秒并多次取平均值。
*   **消融实验**：系统性地测试了标准 OS 优化手段（大页内存、核心绑定 CPU Pinning、Intel CAT 缓存分区）的局限性，证明了传统方法无法解决 CPU 路径瓶颈。
*   **充分性**：实验设计涵盖了延迟（TTFT, TPOT, ITL）、吞吐量、能效比以及不同百分位（P50, P99, P99.9）的尾部延迟，数据详实，对比公平（尤其是与使用相同引擎的 TRT-LLM 对比）。

#### 6. 主要结论与发现
*   **性能提升**：在隔离环境下，Blink 将 P99 首字延迟（TTFT）降低了高达 **8.47 倍**，解码吞吐量提升了 **2.1 倍**。
*   **干扰免疫**：在 CPU 强干扰下，现有系统性能下降 1-2 个数量级，而 Blink 的性能几乎保持不变（波动在实验误差范围内）。
*   **能效显著**：由于消除了 CPU 轮询和调度开销，单 Token 能耗在隔离时降低 **48.6%**，在干扰环境下降低高达 **70.7%**。
*   **MoE 优势**：对于计算量较小但调度频繁的 MoE 模型，Blink 的架构优势最为明显。

#### 7. 优点
*   **架构创新**：首次实现了端到端的 CPU-Free LLM 推理栈，打破了传统的“主机-设备”控制模型。
*   **强隔离性**：为云服务商提供了真正的硬件级性能隔离，使得 LLM 推理可以与高负载 CPU 任务安全混部。
*   **低延迟**：通过 GPU 内部调度和 RDMA 直连，极大地压缩了非计算性的编排延迟。

#### 8. 不足与局限
*   **单 GPU 限制**：当前原型主要针对单 GPU 场景，虽然文中讨论了扩展到多 GPU 的潜力，但尚未给出多卡并行的实验数据。
*   **显存开销**：为了实现设备端启动，需要预先捕获并缓存大量不同形状的 CUDA Graphs，这会占用额外的 GPU 显存（约 2-4 GB）。
*   **功能完整性**：目前尚未集成 Chunked Prefill（分块预填充）和 Prefix Caching（前缀缓存）等高级优化技术，尽管架构上支持这些扩展。
