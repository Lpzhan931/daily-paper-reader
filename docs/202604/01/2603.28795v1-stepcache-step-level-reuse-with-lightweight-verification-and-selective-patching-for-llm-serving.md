## 摘要
我们针对大语言模型（LLM）推理服务中的工作负载，其中重复请求共享通用的解题结构，但在局部约束（如输出模式、变量名或数值常量）上存在差异。现有的缓存方法通常重用完整响应（语义缓存）或模型内部的 KV/前缀状态，前者在局部变化下较为脆弱，后者则与特定后端紧密耦合。我们提出了 StepCache，这是一个与后端无关的步骤级重用层。它将输出分割为有序步骤，检索匹配度最高的缓存请求，使用轻量级的任务感知检查来验证步骤，并通过选择性修补仅重新生成失败的区域。StepCache 还支持针对 JSON 的严格结构化输出强制执行，包括单步提取、必填键约束和单次修复，以及针对语义变化的保守跳过重用回退机制。对于线性方程，StepCache 通过有界修复循环将验证提升为校正，并配备确定性回退机制，以确保在后端模型失效时仍能保证正确性。在数学和 JSON 变体的纯 CPU 高扰动微基准测试中（三次随机种子平均值），StepCache 将平均延迟从 2.13 秒降低至 0.67 秒，中位数延迟从 2.42 秒降低至 0.01 秒，p95 延迟从 3.38 秒降低至 3.30 秒。它还将总 Token 使用量从 36.1k 减少到 27.3k，并在任务特定检查和拼接输出完整性检查下，将端到端正确率从 72.5% 提高到 100%。在所有请求中，79.7% 走仅重用的快速路径，5.4% 需要修补，14.9% 触发了跳过重用。

## Abstract
We address LLM serving workloads where repeated requests share a common solution structure but differ in localized constraints, such as output schema, variable names, or numeric constants. Prior caching approaches typically reuse either full responses (semantic caching) or model-internal KV/prefix states, which are respectively brittle under partial changes or tightly coupled to specific backends. We present StepCache, a backend-agnostic step-level reuse layer that segments outputs into ordered steps, retrieves the best-matching cached request, verifies steps using lightweight task-aware checks, and regenerates only failing regions via selective patching. StepCache additionally supports strict structured-output enforcement for JSON, including single-step extraction, required-key constraints, and one-shot repair, as well as conservative skip-reuse fallbacks for semantic changes. For linear equations, StepCache promotes verification into correction via a bounded repair loop with a deterministic fallback that guarantees correctness when the backend model fails.   In a CPU-only perturbation-heavy micro-benchmark on math and JSON variants, averaged over three seeds, StepCache reduces mean latency from 2.13 s to 0.67 s, median latency from 2.42 s to 0.01 s, and p95 latency from 3.38 s to 3.30 s. It also reduces total token usage from 36.1k to 27.3k and improves end-to-end correctness from 72.5% to 100% under task-specific checks and a stitched-output integrity check. Across requests, 79.7% take the reuse-only fast path, 5.4% require patching, and 14.9% trigger skip-reuse.

### 1. 核心问题与研究动机
*   **核心问题**：现有的 LLM 推理缓存技术存在局限性。**语义缓存（Semantic Caching）**通常是“全或无”的重用，面对局部约束变化（如修改 JSON 键名或数学常数）时容易失效；**前缀/KV 缓存**则与特定模型后端、分词器紧密耦合，且主要加速前缀重用，无法处理答案中间部分的重用。
*   **研究动机**：在实际应用（如代码助手、智能体工作流）中，许多请求具有稳定的解题结构，仅在局部细节上有所不同。作者旨在提出一种**与后端无关（Backend-agnostic）**、**细粒度（步骤级）**的重用机制，以降低延迟、节省 Token 并提高正确性。

### 2. 方法论
StepCache 的核心思想是将 LLM 的输出视为有序的“步骤”序列，并引入验证与修补机制：
*   **步骤级抽象（Step-level Abstraction）**：将响应分割为多个步骤（如按段落、枚举项或 JSON 对象分割）。
*   **检索与匹配**：使用向量嵌入（Sentence-BERT）计算提示词相似度，从缓存中检索最匹配的先前请求。
*   **轻量级验证（Lightweight Verification）**：
    *   **数学任务**：检查中间等式和最终结果是否符合当前提示词的数值逻辑。
    *   **JSON 任务**：验证语法正确性及是否包含所有必填键。
*   **选择性修补（Selective Patching）**：
    *   **连续块修补**：若某一步验证失败，则从该步开始及之后的所有步骤进行重新生成，以维持逻辑依赖。
    *   **结构化修补**：针对 JSON 任务，使用专用提示词强制生成符合模式的输出。
*   **自适应策略**：
    *   **保守跳过（Skip-reuse）**：当检测到核心语义变化（如数学常数改变）或失效步骤过多时，直接回退到完整生成。
    *   **确定性回退**：对于数学任务，若多次修复失败，则使用确定性算法计算结果作为最终保障。

### 7. 优点
*   **后端无关性**：不依赖模型内部状态（如 KV Cache），可作为插件置于任何 LLM API（如 vLLM, OpenAI）前端。
*   **细粒度灵活性**：比传统的语义缓存更灵活，能够处理“结构相似但细节不同”的请求。
*   **可靠性保障**：引入了任务感知的验证和确定性回退，解决了缓存可能带来的幻觉或陈旧数据问题。

### 8. 不足与局限
*   **验证成本**：对于数学和 JSON 等有明确规则的任务，验证很轻量；但对于开放式文本生成，验证可能需要调用“LLM 裁判”，这会增加额外开销。
*   **依赖处理较为保守**：目前的“连续块修补”策略在某一步失败后会重新生成后续所有内容，可能仍存在一定的计算浪费。
*   **应用范围限制**：目前仅在结构化较强的任务上表现优异，在创意写作或长篇对话等非结构化场景下的表现尚待观察。
